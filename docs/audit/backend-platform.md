# backend-platform

## Domain summary
apps/api is a NestJS 10 + Prisma 5 backend; datasource is SQLite (apps/api/prisma/schema.prisma:6, DATABASE_URL env) with docker-compose Postgres offered only as an optional production switch. Entry point apps/api/src/main.ts sets global prefix /api, CORS from CORS_ORIGIN (comma-split, default http://localhost:5173, credentials:true), and a global ValidationPipe (whitelist+transform). app.module.ts wires 17 feature modules plus a @Global PrismaModule (prisma/prisma.service.ts extends PrismaClient, $connect on init, no $disconnect/shutdown hooks). Auth: auth/ has login (POST /auth/login) with bcryptjs + constant-time dummy-hash compare, GET/PUT /auth/me; JwtModule.registerAsync signs HS256 tokens with secret from common/jwt-secret.ts (fails fast if <16 chars — good), expiry JWT_EXPIRES_IN default 7d. JWT payload embeds sub/username/role/name/agentId; jwt.strategy.ts trusts the payload verbatim with no DB lookup. RolesGuard (auth/roles.guard.ts) is default-allow when no @Roles metadata; there is no global APP_GUARD — every controller opts in with @UseGuards(JwtAuthGuard, RolesGuard) (all 16 controllers do). Roles are free-text strings on User.role (ADMIN | ACCOUNTANT | AGENT | CASHIER per schema comment; no enum). Domain modules follow one pattern: thin controller (@Body() d: any — no DTOs anywhere except auth/dto.ts), service computing party balances two duplicated ways: findAll via prisma groupBy + Map join; findOne via in-memory reduce over the entity's full orders/payments arrays. Shared helpers live in common/recognition.ts: RECOGNIZED_ORDER (status != CANCELLED — debt recognized at order creation per owner decision), roundMoney, and recognizedPaymentWhere which materializes ALL cancelled order ids into a notIn list per call. clients service scopes AGENT users by token agentId for findAll/create only; dashboard service scopes all four endpoints for AGENT; agents/factories/vehicles/products controllers leave GET routes role-unrestricted (any authenticated role). reports/svod and dashboard mostly use SQL aggregates (good), except dashboard.salesTrend which loads every order row and aggregates by day in JS. debts.service.summary (used by dashboard) loads full client/factory/vehicle tables plus 6 groupBys. import/ exposes POST /import/excel (ADMIN/ACCOUNTANT) using multer FileInterceptor with no size/type limits and xlsx@0.18.5; importWorkbook runs one interactive transaction (120s timeout), optionally wiping payments/orders/products/clients/expenses/cash mirrors (replace=true), then parses three hardcoded sheets (Tovar/Оплата/Оплата завод) by fixed column indices, hardcoded factory 'CAOLS KS' and region 'Xorazm Beruniy', get-or-create caches for agents/clients/products/vehicles, per-row try/catch that counts silent skips, and mirrors payments into cashTransaction rows. No pagination exists on any list endpoint; no rate limiting (@nestjs/throttler absent); no helmet. users/ is ADMIN-only CRUD with default password 'smartblok'; agents.create auto-provisions an AGENT login with default password 'agent123'. Client.name, Agent.name, Factory.name, Region.name are all @unique in schema; P2002 handled only in products.service.

## Findings
### [critical/CONFIRMED] AGENT role can read company-wide financials of all agents (missing @Roles on GET routes)
apps/api/src/agents/agents.controller.ts:11
GET /agents and GET /agents/:id have no @Roles decorator; RolesGuard (auth/roles.guard.ts:14) returns true when no roles metadata exists, so ANY authenticated user — including AGENT and CASHIER — can call them. agents.service.findAll (agents.service.ts:10-28) returns every agent with sales, profit, and collected sums; findOne (lines 30-58) returns any rival agent's full client list (names, phones), all their orders with per-unit cost/sale prices and profit, and outstanding/advance totals. This directly contradicts the deliberate scoping done in dashboard.controller.ts:8-12 ('nothing global leaks to a low-privilege login') — an agent just calls /api/agents instead of /api/dashboard/agent-performance.
EVIDENCE: 1) Missing @Roles on GET routes — apps/api/src/agents/agents.controller.ts:7-15: the class has only `@UseGuards(JwtAuthGuard, RolesGuard)` (line 7, no class-level @Roles); line 11: `@Get() findAll() { return this.service.findAll(); }` and line 12: `@Get(':id') findOne(@Param('id') id: string) { return this.service.findOne(id); }` — neither has @Roles, while the mutating routes do (lines 13-15: `@Roles('ADMIN', 'ACCOUNTANT') @Post()...`, `@Roles('ADMIN') @Delete...`), proving the omission on GETs is not covered elsewhere in the class.

2) Guard allows when metadata absent — apps/api/src/auth/roles.guard.ts:14: `if (!required || required.length === 0) return true;` — so any authenticated user passes. Line 16 checks `user.role`, and jwt.strategy.ts:17-24 attaches `role` from the JWT payload; schema.prisma:16 confirms low-privilege roles exist: `role String @default("AGENT") // ADMIN | ACCOUNTANT | AGENT | CASHIER`.

3) Data exposure as claimed — apps/api/src/agents/agents.service.ts:22-27 (findAll): returns every agent spread with `sales`, `profit`, `collected` sums plus linked usernames (line 14 `users: { select: { id: true, username: true } }`). findOne (lines 30-58): line 33 `include: { clients: { orderBy: { name: 'asc' } }, ... }` returns the full Client rows (names/phones) of any agent by id; line 41 returns all their orders `include: { client: true, product: true, factory: true, vehicle: true }` — Order rows carry `costPricePerUnit`, `salePricePerUnit`, `costTotal`, `saleTotal`, `profit` (prisma/schema.prisma:147-152); line 58 returns `totals: { sales, profit, collected, outstanding, advance, ... }`.

4) No mitigation elsewhere — grep for `APP_GUARD|APP_INTERCEPTOR|useGlobalGuards|MiddlewareConsumer` across apps/api/src returned no matches; agents.module.ts:4 registers the controller as-is. Neither findAll nor findOne takes a user/agentId argument, so no service-level scoping exists.

5) Contradiction with dashboard scoping is verbatim — apps/api/src/dashboard/dashboard.controller.ts:8-12: comment 'Previously guarded only by JwtAuthGuard — any authenticated user could read company-wide financials and every agent's numbers. Now role-guarded, and the service scopes an AGENT to their own agentId ... so nothing global leaks to a low-privilege login.' followed by `@Roles('ADMIN', 'ACCOUNTANT', 'AGENT')`. An AGENT (or CASHIER) token calling GET /agents or GET /agents/:id bypasses exactly that protection.
REC: Add @Roles('ADMIN','ACCOUNTANT') to findAll/findOne, or scope: an AGENT may fetch only their own agent record (compare user.agentId to :id, 403 otherwise).

### [critical/CONFIRMED] IDOR: AGENT can read and UPDATE any client of any agent, including raising creditLimit

apps/api/src/clients/clients.service.ts:30-40 — `findOne(id)` does `this.prisma.client.findUnique({ where: { id } ... })` with NO agent scoping (the `scope()` helper at lines 9-11 is applied only in findAll, line 14), then returns `{ ...client, orders, payments, totals }` including full order and payment history (lines 33-40). Bypass impact confirmed: apps/api/src/orders/orders.service.ts:130-133 — `// credit limit (creditLimit 0 = unlimited)` / `if (client.creditLimit && client.creditLimit > 0 ...)` — the enforcement reads client.creditLimit, so an AGENT PUT of a huge value (or even `creditLimit: 0`, which the code treats as UNLIMITED) fully disables the owner's credit-limit rule.

### [high/CONFIRMED] JWT never re-validated against DB: deactivated users keep access up to 7 days; stale roles

JwtStrategy.validate() (lines 17-25) returns the token payload verbatim — no lookup of the user row. ### [high/CONFIRMED] recognizedPaymentWhere materializes all cancelled-order IDs into notIn — breaks at scale, re-queried repeatedly
apps/api/src/common/recognition.ts:21
recognizedPaymentWhere fetches every CANCELLED order id (lines 21-24) and builds `orderId: { notIn: ids }` (line 27). It is also re-executed per call: debts.summary runs it 3x (debts.service.ts:10-14), dashboard.summary adds another, agents.findOne 2x (agents.service.ts:38-39) — identical full scans of the orders table per HTTP request.
EVIDENCE: recognition.ts:21-24: `const cancelled = await prisma.order.findMany({ where: { status: 'CANCELLED' }, select: { id: true } });` and recognition.ts:27: `return { AND: [base, { OR: [{ orderId: null }, { orderId: { notIn: ids } }] }] };` — every cancelled order id materialized into an unbounded notIn list, no chunking anywhere; the only early-out is recognition.ts:25 `if (!cancelled.length) return base;`. schema.prisma:6: `provider = "sqlite"` — SQLite confirmed, and Prisma does not chunk user-supplied notIn filters, so the query throws once the list exceeds SQLite's bind-variable ceiling (32766 in Prisma 5's bundled SQLite; the auditor's 999/32766 hedge is accurate). Soft-cancel-only deletion is even documented in recognition.ts:6: "CANCELLED orders (including soft-cancelled deletions) contribute nothing" — the list only grows. Repeat execution confirmed: debts.service.ts:11-13 (three calls: type CLIENT/FACTORY/VEHICLE inside summary()), dashboard.service.ts:51 (`const payCliWhere = await recognizedPaymentWhere(...)` in summary(), plus :32 and :117), agents.service.ts:38-39 (two calls in findOne). CORRECTED: One wording nuance: the repeated query is not a "full scan of the orders table" — schema.prisma:163 has `@@index([status])` on Order, so each call is an index-backed fetch of only the CANCELLED rows. ### [high/CONFIRMED] Import date parsing transposes day/month or silently substitutes today's date
apps/api/src/import/import.service.ts:13
toDate() (lines 10-15) parses string dates with `new Date(v)`. The workbook is Russian/Uzbek (dd.mm.yyyy): V8 parses '03.04.2025' US-style as March 4 (day/month swapped) whenever day<=12, and returns Invalid for day>12, which falls through to `return new Date()` (line 14) — the row is imported dated TODAY. Historical orders/payments get silently wrong dates, corrupting date-based reporting and debt aging. Compounding this, every row-level failure is swallowed by bare `catch { result.skipped++ }` (lines 151, 179, 196) with no row numbers or reasons returned, so operators cannot tell what was dropped or mis-dated.
EVIDENCE: apps/api/src/import/import.service.ts:10-15 — `function toDate(v: any): Date { if (v instanceof Date && !isNaN(v.getTime())) return v; if (typeof v === 'number') return new Date(Math.round((v - 25569) * 86400 * 1000)); if (typeof v === 'string' && v.trim()) { const d = new Date(v); if (!isNaN(d.getTime())) return d; } return new Date(); }` — line 13 parses strings with `new Date(v)`, line 14 falls back to today for unparseable strings AND for null cells. Mitigating nuance (does not refute): line 33 `XLSX.read(buffer, { cellDates: true })` plus `raw: true` (line 28) means genuinely date-typed Excel cells arrive as Date objects and are handled correctly at line 11 — I parsed the bundled reference workbook docs/Газоблок Счет.xlsx and all 56 importable Товар rows and 25 Оплата rows have Date-typed date cells, so this particular file does not currently exercise the transposition/today-fallback. The defect is therefore latent for the shipped sample but unconditionally present in code and fires on any uploaded workbook where dates are stored as text (dd.mm.yyyy) or left blank/malformed, with no warning surfaced to the operator.

### [medium] Import merges same-named clients across agents and ignores the sheet's agent on existing clients

getClient (lines 71-78) looks up by name only (`tx.client.findFirst({ where: { name } })`) and caches by lowercased name; agentId is used only on creation. ### [medium] 120s single transaction import blocks the entire SQLite database
apps/api/src/import/import.service.ts:201
importWorkbook runs one interactive transaction with sequential per-row awaited creates (agents, clients, products, vehicles, orders, payments, cash mirrors — several queries per row) and timeout/maxWait of 120000ms (line 201). On SQLite (schema.prisma:6) a write transaction locks the whole database file, so every other API request (orders, payments, dashboard) stalls or errors for up to two minutes during an import. Large workbooks exceeding 120s abort entirely.
REC: Batch with createMany where possible, chunk the import into multiple transactions with a resumable job record, or run imports against a staging table; at minimum document the freeze and run imports off-hours.
