# backend-financial

## Domain summary
SmartBlok API (NestJS 10 + Prisma 5 + SQLite, all money fields Float) implements a derive-don't-store debt model: no Debt table; client/factory/vehicle balances are recomputed on every request from Order and Payment aggregates. apps/api/src/common/recognition.ts is the single source of recognition truth: RECOGNIZED_ORDER = {status:{not:'CANCELLED'}} (debt recognized at ORDER CREATION, matching the owner decision), roundMoney = Math.round, and recognizedPaymentWhere() which excludes payments linked to CANCELLED orders by fetching all cancelled order ids into a notIn list. This constant is consistently imported by orders, debts, clients, factories, vehicles, agents, dashboard, and reports services — no recognition-timing drift between modules. orders.service.ts computes totals server-side (costTotal=q*cost, saleTotal=q*sale, profit=sale-cost-transportFee) with per-unit prices defaulting from Product but overridable by the caller; credit limit (Client.creditLimit, 0=unlimited) is enforced only in create(); deletion is soft-cancel (status=CANCELLED); orderNo is B-NNNN derived from max existing suffix with a 5-retry P2002 loop. payments.service.ts posts Payment + CashTransaction atomically in $transaction; method→cashbox mapping is by hardcoded seeded cashbox names; USD payments compute amount=usdAmount*rate and write usdAmount to the USD box, UZS amount elsewhere; delete is hard-delete + kassa row deleteMany (also transactional). expenses.service.ts mirrors this (UZS-only boxes enforced). kassa.service.ts derives box balances from IN-OUT groupBy sums; manual transactions validated (amount>0, USD box requires rate); PAYMENT/EXPENSE-sourced rows cannot be deleted from the journal. debts.service.ts computes three-party balances (client saleTotal, factory costTotal, vehicle transportFee vs typed payments) with per-party clamped totals. procurement/ is pure analytics: landedCostPerM3 = pricePerM3 + costPerTruck/truckCapacityM3 (default capacity 33 hardcoded), netCostAfterBonus = landed*(1-dealerBonusPct) expecting a 0..1 fraction. dashboard.service.ts delegates debt totals to DebtsService and splits cash by currency. Controllers use JWT + role guards (ADMIN/ACCOUNTANT/AGENT/CASHIER) with agent scoping, but ALL bodies are typed `any` — there is no ValidationPipe/class-validator anywhere. docs/05-biznes-jarayonlari-va-formulalar.md (TZ v1.0) is materially STALE versus code: it documents recognition at DELIVERED/COMPLETED (code: at creation), creditLimit as unenforced (code: enforced in create), count()-based orderNo (code: max-based), hard-delete remove() that detaches payments (code: soft-cancel), and import not posting to kassa (import.service.ts does mirror kassa entries). The main defect clusters found: unvalidated/negative/non-finite money inputs with caller-controlled prices, credit-limit bypasses (update, un-cancel, TOCTOU), AGENT privilege bypass via setStatus(CANCELLED), orphaned cash on cancellation with no refund flow, unvalidated payment orderId enabling balance-invisible payments, unbounded FX rate, no idempotency, hard-deleted payment history, and a scalability landmine in recognizedPaymentWhere.

## Findings
### [critical/PARTIAL] Order money inputs unvalidated: negative/zero/NaN/Infinity quantity, prices and transportFee accepted; enables stealth debt reduction and credit-limit bypass
apps/api/src/orders/orders.service.ts:10
totals() (lines 10-18) coerces via Number(x)||0 with no sign/finiteness/zero checks, and numOr (21-23) passes any caller value through. Concrete failures: (a) POST /orders with quantity=-100 stores negative saleTotal, which REDUCES the client's derived balance (clientBalance line 81 and debts.service.ts:34) — an agent can erase a client's debt by booking a negative order; (b) negative saleTotal also passes the credit-limit check at line 133 (current + negative <= limit); (c) salePricePerUnit='abc' → NaN||0 → 0 → goods booked for free, silently; (d) quantity='Infinity' → Infinity propagates into saleTotal/profit and poisons every aggregate (debts, dashboard, reports); (e) negative transportFee inflates profit (line 17). Same paths exist in update() via totals({...existing,...dto}) at line 193. There is no ValidationPipe/DTO class anywhere in the API (all controllers take @Body() d: any).
EVIDENCE: All five concrete failure modes CONFIRMED by direct code reading; only the final supporting sentence ("no ValidationPipe/DTO class anywhere") is wrong. Evidence: (1) apps/api/src/orders/orders.service.ts:10-18 totals(): `const quantity = Number(d.quantity) || 0; ... const saleTotal = quantity * sale; ... profit: saleTotal - costTotal - transport` — no sign/zero/finiteness checks; negative quantity → negative saleTotal, negative transportFee inflates profit (line 17). (3) Credit-limit check orders.service.ts:133: `if (current + t.saleTotal > client.creditLimit)` — negative saleTotal always passes; and update() (lines 169-207) recomputes `totals({ ...existing, ...dto, ...priceDefaults })` at line 193 with NO credit-limit check at all. (4) Balance derivation: orders.service.ts:78-81 clientBalance sums saleTotal (`(ordAgg._sum.saleTotal ?? 0) - (payAgg._sum.amount ?? 0)`), and apps/api/src/debts/debts.service.ts:33-36 `const delivered = ocM.get(c.id) ?? 0 ... balance: roundMoney(delivered - paid)` — a negative-quantity order directly reduces the client's displayed debt. (5) AGENT role can POST /orders: orders.controller.ts:8-9 `@Roles('ADMIN','ACCOUNTANT','AGENT')`, line 15 `@Post() create(...@Body() d: any)`; the agent-scope check (service lines 110-115) only restricts WHICH client, not values — so an agent can erase their own client's debt with a negative order. ### [critical/CONFIRMED] Credit limit enforced only in create(): fully bypassable via update(), via un-cancel through setStatus, and racy (TOCTOU) even in create
apps/api/src/orders/orders.service.ts:131
The owner's rule is 'credit limits must be enforced'. The check exists only at create() lines 131-138. Bypasses: (1) update() (169-207) recomputes totals with no credit re-check — create an order for quantity 1, then PUT quantity=10000; (2) update() also allows changing clientId (line 199) with no limit check on the target client and no existence/ownership validation of the new client; (3) setStatus() (209-216) allows CANCELLED→NEW/COMPLETED (VALID_STATUS includes every value, no transition matrix), resurrecting a cancelled order's debt with no re-check — and a caller can create the order as status CANCELLED (skipping the check via `status !== 'CANCELLED'` at 131) then flip it live; (4) TOCTOU: clientBalance() at line 132 and order.create at 156 are not in one transaction, so two concurrent orders both pass the check and jointly blow the cap.
EVIDENCE: All four bypass vectors verified in apps/api/src/orders/orders.service.ts; grep for `creditLimit` across apps/api/src shows the check exists ONLY in create() (plus mere storage in clients.service.ts), and grep for `$transaction` shows orders.service.ts uses none.

Check exists only in create(): lines 131-133: `if (client.creditLimit && client.creditLimit > 0 && status !== 'CANCELLED') { const current = await this.clientBalance(dto.clientId); if (current + t.saleTotal > client.creditLimit) {...`. update() (lines 169-207) recomputes totals at line 193: `const t = totals({ ...existing, ...dto, ...priceDefaults });` and writes them at 194-206 (`...t`) with zero credit-limit or clientBalance code. No DTO mitigation: controller line 16 is `@Put(':id') update(... @Body() d: any)` — the global ValidationPipe in main.ts (lines 13-19) has no effect because handlers type bodies as `any` (no DTO class, so whitelist/validation never runs). Controller class-level `@Roles('ADMIN', 'ACCOUNTANT', 'AGENT')` (line 9) means even AGENTs can do this.

(2) clientId swap — CONFIRMED. Line 199: `...(dto.clientId !== undefined ? { clientId: dto.clientId } : {}),` — no findUnique on the target client, no credit check against the target client's limit, no check that the new client belongs to the requesting AGENT (assertOwnership at 172 only checks the EXISTING order's agentId). (3) setStatus resurrection — CONFIRMED. No $transaction/locking anywhere in the file (grep: $transaction hits only payments/agents/import/expenses services). Two concurrent creates each read the pre-insert balance and both commit. The P2002 retry loop (154-165) re-runs only the insert, never the credit check.

All cited line numbers in the claim are accurate (131-138, 169-207, 199, 209-216, 132, 156).
REC: Extract a checkCreditLimit(clientId, deltaSale) helper; call it in update() (on saleTotal delta and on clientId change) and in setStatus/advance when leaving CANCELLED. Wrap balance-check + create in prisma.$transaction (SQLite serializes writes, making the check-then-insert atomic).

### [medium] recognizedPaymentWhere builds an unbounded notIn list of all cancelled order ids — breaks outright at SQLite variable limits and is O(cancelled) on every balance read
