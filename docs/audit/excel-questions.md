# Open questions
- Pallet economics: clients are never billed money for pallets (Сумма Продажа = m³ × price only) but owe them in kind (K5−E5). Confirm: (a) does the factory refund 130 000 UZS per pallet the dealer returns, i.e. is Завод Остаток 'с паддон' (B4) or 'без паддон' (B5) the real factory balance? (b) what happens if a client never returns pallets — charge 130k each, and after how long?
- Payment source of truth: the workbook's own фарк (Свод B20 = −95 831 233) shows client sheets contain ~95.8M of payments absent from the Оплата ledger (Шиддат ~63M, Ирригатсия 32.832M, Уткир 1.964M). Which record wins for migration — client sheets or the Оплата ledger — and should those missing payments be entered into the ERP as real receipts (which cashbox/legal entity received them)?
- Client-paid transport ('шопр учун барди' / 'клентдан', 27.5M total): confirm the intended double effect — credit the client's account AND settle that truck's transport liability, with no money entering the dealer's kassa. Should the ERP model this as a dedicated payment type, and should such amounts count in agent collection KPIs?
- Dealer-paid transport (Расход Авто, 68.1M, status column 'Туланди'/date/blank): should import create matching VEHICLE debts and payments so trucks marked 'Туланди' show zero balance, instead of leaving 68.1M of phantom vehicle debt?
- Profit definition: the workbook's Общая прибль = sale − block cost (ignores transport and pallets); the ERP computes profit = saleTotal − costTotal − transportFee. Which definition should reports show, and should unrecovered pallet cost ever hit profit?
- Unpriced deliveries (Шиддат моналит trucks with no Цена Продажа, client meanwhile prepaid 86M): can goods ship before a price is agreed, and if debt is recognized at order creation (owner rule), at what price is it recognized — provisional, last price, or zero until priced?
- Lump-sum pricing: several sale prices are back-computed from negotiated totals (732 542.438, 729 928.1 per m³). Should order entry accept a lump-sum saleTotal and derive the unit price (keeping totals exact), and is a sub-1-UZS residual balance (e.g. Рустам Шпик +0.62) considered settled?
- Cube/pallet inconsistencies: 23.4 m³ trucks appear with 13 pallets (1.8 m³/pallet, size 600x300x250) but also with 19 pallets (mini trucks?). Provide the authoritative pallet-volume table per block size so the ERP can validate m³ = pallets × volume.
- Client identity: confirm canonical names/merge list for spelling variants (Жасур/Жаср Версал; Шиддат моналит/маналит; Нормат Умидбек/NORMAT UMIDBEK; Гофур Хазорасп/хазорасп) and clients without account sheets (Отабек дамирчи). Also resolve the Уткир мини conflict (Товар: two 23.4 m³ trucks, plate 95 653 GBA vs client sheet: one 32.832 m³ truck, plate 95 693 GBA).
- Agents: Темур and Сардор ога both carry number 5 (Сардор's client sheet is prefixed 6). Are agent numbers meaningful (commission, territory) or just sheet ordering? Is there any agent commission/bonus logic at all — none is visible in the workbook?
- Multiple legal entities: factory payments go out via 'Септем Алока', cash ('Нахт'), and card ('Нахт пластика'), and some client money arrives at 'Септем семент'. Should the ERP model separate cashboxes/legal entities per payer/receiver, and how do inter-entity balances settle?
- Cash columns in Оплата (Накд + denomination columns −5/−10/−15/−20/−50 + Сумма): confirm these are banknote-count fields and how a cash payment will actually be recorded, since the current import method-detection reads only Накд (r[5]) and would classify a denomination-only row as BANK.
- Factory payment methods: 'Нахт' and 'Нахт пластика' rows are cash, but import hardcodes method=BANK for all factory payments, pulling cash out of the Bank kassa. Confirm correct method mapping and which cashbox each factory payment should debit.
- Debt recognition timing: owner rule says client debt at ORDER CREATION, workbook recognizes at shipment row entry, current code/docs at DELIVERED/COMPLETED. Confirm the single rule for the new ERP (and how credit-limit checks interact with unpriced or partially-priced orders).

# Doc discrepancies
- docs/05 §5.3.2/§5.11 defines profit = saleTotal − costTotal − transportFee; the workbook's Общая прибль (Товар col T) = СуммаПродажа − СуммаПриход, ignoring transport entirely (and 5 legacy Разница rows even add pallet price). The two profit figures differ by Σ transport (68.1M).
- docs/05 §5.6 factory balance = Σ costTotal − Σ FACTORY payments (blocks only); the workbook's headline Завод Остаток (Свод B4) = payments − (blocks + pallets), i.e. includes 135.2M of pallet purchases (Блок+Поддон, Товар col N). The docs and code have no pallet concept at all — Поддон Шт/Цена Поддон/Сумма Поддон, per-client pallet balances (K5−E5), and Возврат паддон returns are an entire workbook subsystem absent from docs 05 and 09.
- docs/05 and 09 nowhere mention the 'Клент шопрга барди'/'шопр учун барди' mechanism: client-paid driver amounts (27.5M) sit inside client-sheet payment columns and as text 'клентдан' in Товар col S (r[18]); import's num() converts 'клентдан'/'Х' to 0, losing both the transport cost and the client credit — after import client debts are overstated and the semantics are gone.
- docs/09 §9.4 claims the three sheets are the complete data source; the workbook's own reconciliation rows (Свод B20 фарк = −95 831 233) prove the Оплата ledger is missing ~95.8M of payments that exist only in the 17 client sheets (Шиддат ~63M, Ирригатсия 32.832M, Уткир 1.964M). Import per docs/09 therefore understates client payments and overstates client debt by that amount; client sheets and Свод Завод are not imported at all.
- docs/09 §9.4.2 says the Oplata loop skips '1 header row' (i=1); in the actual workbook the used range starts at Excel row 3, so index 0 is the SUBTOTAL row and index 1 is the HEADER row — the header row is processed as data, creating junk catalog entries (agent 'Агент', client 'Клиент') before the zero-amount check skips the payment (verified against import.service.ts: getAgent/getClient run before the amount guard).
- docs/09 §9.4.3 maps Оплата Завод r[3] as 'izoh (note)'; in the workbook column D is 'Получател' (receiver — the factory legal entity or a card number 9860190103054400), not a note.
- docs/09 §9.8.3 states factory payments are always method='BANK'; the workbook has cash payers 'Нахт' and 'Нахт пластика' — with the current code these cash payments are posted OUT of 'Bank kassa', misstating both kassas.
- docs/09 §9.1/§9.6/§9.9 and docs/05 §5.12(12) state the import creates NO CashTransaction rows, replace is NOT in a transaction, and replace leaves orphaned kassa rows; the current import.service.ts contradicts all three: the whole import runs in one $transaction, replace also deletes cashTransaction(source PAYMENT/EXPENSE) and expense rows, and makePayment() creates mirror CashTransaction entries for every imported payment.
- docs/09 §9.8.1 says the orderNo counter starts from order.count(); current code derives it from the maximum numeric suffix of existing orderNo values (gap-safe).
- docs/09 §9.5.2/§9.11 claims name-based find-or-create prevents duplicate catalog records; the workbook uses divergent spellings for the same client across sheets ('Жасур Версал' vs 'Жаср Версал', 'Шиддат моналит' vs 'Шиддат маналит', 'Нормат Умидбек' vs Latin 'NORMAT UMIDBEK') — import will create duplicate clients and split one client's orders and payments across records.
- docs/09 §9.4.1 ignores Товар col U 'Авто услу барлдми?' (transport paid: 'Туланди'/date) and creates no VEHICLE payments; after import the ERP reports Σ transportFee (68.1M) as open vehicle debt although the workbook marks most trucks paid.
- docs/09 §9.4.2 method detection reads only r[5] Накд for CASH; the workbook's Оплата also has denomination columns G–K (−5…−50) and a computed 'Сумма' P=r[15] that feed the row total (Жами = ПР-Сумма + Сумма + Прочие) — a cash payment recorded via denominations would be silently classified as BANK.
- docs/05 §5.6.1 sign convention (positive balance = client owes us) is inverted relative to the workbook (Остаток F2 = payments − goods, negative = client owes); migration validation must flip signs when comparing, and docs never state this mapping.
- docs/05 §5.2 ties debt recognition to DELIVERED/COMPLETED status; the workbook has no status — a shipment row affects the client account immediately — and the recorded owner decision is recognition at order creation; the docs match neither.
