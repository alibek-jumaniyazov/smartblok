# realtime-perf

## Domain summary
SmartBlok monorepo: apps/api = NestJS 10 + Prisma 5 on SQLite (schema.prisma datasource provider="sqlite"; docker-compose.yml ships an OPTIONAL Postgres 16 with a comment saying dev default is SQLite and switching requires editing the provider). apps/web = React 18 + Vite 6 + TanStack React Query 5 + axios + react-router 6 + recharts + framer-motion + Tailwind 4.

REALTIME: none exists. Grep across apps/ for websocket|socket.io|@nestjs/websockets|EventSource|SSE|refetchInterval|setInterval|polling returned zero matches. Neither package.json has any realtime dependency (api: @nestjs/platform-express only, no platform-socket.io; web: no socket client). Freshness relies entirely on React Query defaults in apps/web/src/main.tsx:16 (staleTime 30s, refetchOnWindowFocus:false, no polling) plus blanket cache invalidation after the current user's own mutations. Changes by other users (agents/cashier/admin work concurrently in this ERP) are invisible until navigation remounts a query past its 30s staleTime.

BACKEND QUERY PATTERNS: All money balances are derived at read time (no stored balances). The core recognition helper apps/api/src/common/recognition.ts implements "recognized payment" by fetching ALL cancelled order IDs into JS and building an orderId notIn list — executed per call, and most read endpoints call it 1–3 times (dashboard summary path triggers it 4x including debts.summary). Aggregations are mostly proper prisma groupBy/aggregate (good), but salesTrend, client/factory/agent/vehicle detail endpoints reduce over full findMany result sets in JS. List endpoints (orders, payments, clients, expenses, procurement) are unbounded findMany with heavy includes — only kassa transactions has take:200. Order number generation scans every order row on every create (orders.service.ts nextOrderNo, duplicated in import.service.ts:99). Prisma schema is otherwise well-indexed: FKs on Order/Payment/CashTransaction all have single-column indexes plus Order @@index([status]) and @@index([date]); gaps are Payment/Expense missing @@index([date]) (both list endpoints sort date desc over unbounded scans) and no composite (status,date)/(agentId,date) for the hot order-list filters. All money columns are Float (correctness concern flagged elsewhere; perf-neutral).

FRONTEND: No code splitting — App.tsx statically imports all 21 pages; zero React.lazy/Suspense in src. No table virtualization, but EntityTable.tsx client-side paginates at pageSize 12 so DOM stays bounded; however it searches/filters/CSV-exports over the FULL dataset downloaded from the unbounded API endpoints. Mutations call qc.invalidateQueries() with no key filter in 13 files (17 occurrences), refetching every mounted query (each an unbounded endpoint) after any single mutation. Modest useMemo usage exists (EntityTable filter, NewOrder selectors, Kassa); no React.memo on row components; framer-motion animates each table row (only 12/page, acceptable).

Hot spots read fully: dashboard.service.ts (summary/salesTrend/agentPerformance/orderFunnel — mostly groupBy-based, good, except salesTrend and the recognizedPaymentWhere multiplier), reports.service.ts (svod is clean groupBy/aggregate work), debts.service.ts (9 parallel queries, 3 recognizedPaymentWhere scans), orders.service.ts, payments.service.ts, clients/factories/agents/kassa services.

## Findings
### [high/CONFIRMED] recognizedPaymentWhere loads all cancelled order IDs into a notIn list on nearly every read endpoint
apps/api/src/common/recognition.ts:21
recognizedPaymentWhere() runs prisma.order.findMany({ where: { status: 'CANCELLED' }, select: { id: true } }) and builds { orderId: { notIn: ids } }. It executes on almost every read path, often multiple times per request: dashboard summary calls it directly (dashboard.service.ts:51) and again 3x via debts.summary (debts.service.ts:10-14); reports.svod 2x (reports.service.ts:10-13); clients/agents/factories findAll each 1-2x; orders.create calls it inside the credit-limit check (orders.service.ts:76). Cost grows O(cancelled orders) per invocation, and since deletion = soft-cancel (owner decision), the cancelled set only ever grows. Beyond latency, a large notIn parameter list can hard-fail: SQLite bind-variable limits (999 on older builds, 32766 since 3.32) and Postgres's 65535-param limit mean a few thousand cancelled orders turns every balance/dashboard/report endpoint into a 500, not just a slow query.
EVIDENCE: apps/api/src/common/recognition.ts:21-27 — exactly as claimed: `const cancelled = await prisma.order.findMany({ where: { status: 'CANCELLED' }, select: { id: true } }); if (!cancelled.length) return base; const ids = cancelled.map((c) => c.id); return { AND: [base, { OR: [{ orderId: null }, { orderId: { notIn: ids } }] }] };` — unbounded findMany of ALL cancelled order IDs into an in-memory notIn list; the only mitigation is the empty-set early return, which stops applying after the first soft-cancel.

Call-site fan-out all verified:
- dashboard.service.ts:51 `const payCliWhere = await recognizedPaymentWhere(...)` and dashboard.service.ts:68 `const debts = await this.debts.summary();` — for non-agent users the dashboard summary triggers it 4x total (1 direct + 3 via debts).
- debts.service.ts:10-13 — `Promise.all([recognizedPaymentWhere(..., { type: 'CLIENT' }), recognizedPaymentWhere(..., { type: 'FACTORY' }), recognizedPaymentWhere(..., { type: 'VEHICLE' })])` = 3x per debts summary.
- reports.service.ts:10-13 — 2x per svod (`payAgentWhere`, `payFacWhere`).
- clients.service.ts:15, factories.service.ts:10, vehicles.service.ts:10, agents.service.ts:11 — 1x each in findAll; agents.service.ts:38-39 — 2x in agent detail.
- orders.service.ts:76 (`clientBalance`) invoked from the credit-limit check at orders.service.ts:130-133 (`if (client.creditLimit && client.creditLimit > 0 ...) { const current = await this.clientBalance(dto.clientId);`), so every order creation for a limit-capped client also pays this cost.
- dashboard.service.ts:32 and :117 add two more read-path invocations (agent debt widget, agentPerformance).

DB engine: apps/api/prisma/schema.prisma:6 `provider = "sqlite"` — so the SQLite bind-variable ceiling is the operative hard-fail limit (Prisma auto-chunks large `in` lists in some cases but does not chunk `notIn`, whose chunks cannot be OR-combined, so a sufficiently large cancelled set raises a too-many-SQL-variables engine error rather than degrading gracefully). Grep for cache/memoization across apps/api/src found none related to this path (only unrelated import.service.ts Maps); no middleware, DTO validation, or transaction mitigates it.

Minor nuances (do not overturn the claim): the filter is wrapped as `OR: [{orderId: null}, {orderId: {notIn: ids}}]` (order-less payments always count), and the Postgres 65535-param figure is hypothetical here since the datasource is SQLite. The exact SQLite thresholds (999 / 32766) are engine facts not observable in this repo, but the unbounded notIn construction that makes them reachable was personally observed.
REC: Replace the two-step scan with a relation filter evaluated in SQL: { AND: [base, { OR: [{ orderId: null }, { order: { status: { not: 'CANCELLED' } } }] }] }. This removes the extra roundtrip, the JS materialization, and the parameter-limit failure mode, and makes the function synchronous (callers currently await it needlessly).

### [high/CONFIRMED] nextOrderNo scans every order row on every order creation (up to 5x in the retry loop)
apps/api/src/orders/orders.service.ts:66
nextOrderNo() does prisma.order.findMany({ select: { orderNo: true } }) — the entire orders table — then parses and reduces in JS to find the max numeric suffix. It runs on the hottest write path (order creation) and up to 5 times per create in the P2002 retry loop (line 154-165). O(N) rows transferred per order booked; at 50k orders every new order pulls 50k rows. The same full-scan pattern is duplicated at apps/api/src/import/import.service.ts:99. It is also inherently racy (two concurrent creates compute the same max), which is why the retry loop exists.
EVIDENCE: apps/api/src/orders/orders.service.ts:65-72 — `private async nextOrderNo() { const orders = await this.prisma.order.findMany({ select: { orderNo: true } }); const max = orders.reduce((m, o) => { const n = parseInt(String(o.orderNo).replace(/\D/g, ''), 10); return Number.isFinite(n) && n > m ? n : m; }, 0); return 'B-' + String(max + 1).padStart(4, '0'); }` — findMany with no where/take/orderBy: fetches every order row's orderNo and reduces in JS, exactly as claimed.

apps/api/src/orders/orders.service.ts:152-166 — `// retry on the (rare) concurrent-create orderNo collision` ... `for (let attempt = 0; attempt < 5; attempt++) { try { return await this.prisma.order.create({ data: { orderNo: await this.nextOrderNo(), ...data }, ... }); } catch (e) { lastErr = e; if (e?.code === 'P2002') continue; throw e; } }` — nextOrderNo() is called INSIDE the loop body, so up to 5 full scans per create() on repeated P2002 collisions. create() is the order-creation write path.

apps/api/prisma/schema.prisma:134 — `orderNo String @unique` — confirms the unique constraint that makes the concurrent read-max race manifest as P2002; the code's own comment at line 152 acknowledges the race. No $transaction, advisory lock, or DB sequence exists anywhere in apps/api/src/orders (grep for `\$transaction|advisory|FOR UPDATE|sequence` returned no hits in that dir) — the retry loop is the only mitigation.

apps/api/src/import/import.service.ts:99-103 — `const existingOrders = await tx.order.findMany({ select: { orderNo: true } }); let orderNo = existingOrders.reduce(...)` — the same full-scan/reduce pattern duplicated, as claimed. Minor nuance (not enough to downgrade): in import it runs once per import transaction and increments a local counter, not once per imported order; and the scan transfers only the orderNo column, not full rows — but the claim already quoted the select accurately, and the O(N)-rows-per-order-created characterization of orders.service.ts is exactly right.
REC: Store the numeric part in an Int column (or a single-row counter table updated atomically inside a transaction) and derive orderNo as 'B-' + padded value; alternatively SELECT MAX via a raw query or an indexed orderBy+take(1) on a numeric column. This makes creation O(1) and removes most collision retries.

### [high/CONFIRMED] All main list endpoints are unbounded findMany with heavy includes — no take/skip/server pagination anywhere except kassa
apps/api/src/orders/orders.service.ts:47
orders.findAll returns every order with 5 included relations (agent, client, factory, product, vehicle) — no take/skip; the frontend (Orders.tsx:21) fetches the whole set and paginates client-side at 12 rows (EntityTable.tsx:52). Same unbounded pattern: payments.service.ts:35 (6 includes), clients.service.ts:16, expenses.service.ts:9, procurement.service.ts:23/37/44, and every detail endpoint loads a party's FULL order+payment history (clients.service.ts:34-35, factories.service.ts:35-36, agents.service.ts:41, vehicles.service.ts:29-30). Only kassa.transactions (kassa.service.ts:21) has take:200. For an ERP accumulating years of orders, each page load transfers the entire table (multiplied by the relation payloads), and the blanket invalidateQueries() refetch storm re-downloads them all after any mutation.
EVIDENCE: 1) orders.service.ts:47-50 — `return this.prisma.order.findMany({ where, orderBy: { date: 'desc' }, include: { agent: true, client: true, factory: true, product: true, vehicle: true } });` — 5 includes, no take/skip. Controller (orders.controller.ts:13) passes raw `@Query() q` and findAll only reads status/agentId/clientId/factoryId/vehicleId filters — no pagination params exist.
2) Orders.tsx:21 — `useQuery({ queryKey: ['orders'], queryFn: () => endpoints.orders() })` — fetches full set with no params.
3) EntityTable.tsx:32 `pageSize = 12` default; line 52 `const view = filtered.slice(safePage * pageSize, ...)` — pagination is purely client-side over the full array.
4) payments.service.ts:35-38 — unbounded findMany with 6 includes: `include: { agent: true, client: true, factory: true, vehicle: true, order: true, cashbox: true }`.
5) clients.service.ts:16 unbounded client.findMany; expenses.service.ts:9 unbounded expense.findMany; procurement.service.ts:23 (listPrices), :37 (listRoutes), :44 (matrix prices) all unbounded findMany with includes.
6) Detail endpoints load full histories: clients.service.ts:34-35, factories.service.ts:35-36, agents.service.ts:41, vehicles.service.ts:29-30 — each does `order.findMany({ where: { <partyId> }, include: {3-4 relations} })` plus full payment.findMany, then computes balances by reducing over the full arrays in JS (so the full fetch is load-bearing, not incidental).
7) Grep for `take:|skip:` across apps/api/src returns exactly one hit: kassa.service.ts:21 `take: 200`. No DTO, interceptor, or Prisma middleware adds pagination anywhere else.
8) Blanket refetch storm confirmed: bare `qc.invalidateQueries()` (no key filter — invalidates the entire query cache) appears 17 times across 13 page files, including Orders.tsx:23-24 on every status-advance and delete mutation.
Bonus corroboration: orders.service.ts:66 nextOrderNo() also does a full-table `order.findMany({ select: { orderNo: true } })` on every order creation.
REC: Add server-side pagination (take/skip or cursor) + total count to orders, payments, expenses, and detail-history endpoints; select only needed columns for list views instead of full relation includes; move search filtering server-side so EntityTable can page through the server.

### [high/CONFIRMED] Production-path database is SQLite — single-writer lock is a ceiling for a multi-user ERP
apps/api/prisma/schema.prisma:6
datasource provider is "sqlite"; docker-compose.yml (line 1-3) explicitly says Postgres is optional and requires hand-editing the provider. SQLite serializes all writes behind a single database-level lock — concurrent agents booking orders, cashier recording payments (each payment is a 2-write transaction, payments.service.ts:88-113), and Excel imports (import.service.ts runs one large transaction doing per-row creates) will queue behind each other and can throw SQLITE_BUSY under load. The derived-balance architecture (every dashboard hit = ~15 aggregate queries) amplifies read pressure on the same file. Also groupBy/aggregate performance and the planned enterprise features (reporting windows, locking) assume a real server DB.
EVIDENCE: 1) SQLite is the production-path provider — apps/api/prisma/schema.prisma:5-8: `datasource db { provider = "sqlite"  url = env("DATABASE_URL") }`; apps/api/.env:1: `DATABASE_URL="file:./dev.db"`. 2) Postgres is opt-in and requires hand-editing — docker-compose.yml:1-3: `# Production/optional: PostgreSQL for SmartBlok.` / `# Dev default is SQLite (no setup). To use Postgres, switch the provider in` / `# apps/api/prisma/schema.prisma to "postgresql" and set DATABASE_URL accordingly.` 3) Payment is a 2-write interactive transaction exactly at the cited lines — apps/api/src/payments/payments.service.ts:88-113. 4) Excel import is one giant interactive transaction with per-row awaited creates and a 120s lock window — apps/api/src/import/import.service.ts:38 and :201 `{ timeout: 120000, maxWait: 120000 }`. 5) Dashboard read amplification is undercounted: admin summary() issues ~23 queries per hit (dashboard.service.ts:47-97 + debts.service.ts:10-27 + recognition.ts:21). 6) No mitigations: bare PrismaClient (prisma.service.ts:5-9); zero hits for busy_timeout/WAL/PRAGMA/connection_limit/CacheInterceptor in apps/api.
REC: Commit to Postgres as the default (the compose file already ships it): switch provider, generate migrations, and gate SQLite to local dev only. This also unlocks native Decimal for money and date_trunc-based SQL aggregation for the trend endpoints.

### [medium] No realtime capability at all — multi-user data freshness depends on 30s staleTime and page navigation
apps/web/src/main.tsx:16
Zero realtime infrastructure exists: no websocket/socket.io/SSE/EventSource matches anywhere in apps/, no @nestjs/websockets or platform-socket.io in apps/api/package.json, no socket client in apps/web/package.json, and no polling (no refetchInterval/setInterval in src). React Query is configured with staleTime 30s and refetchOnWindowFocus:false, so a user only sees another user's changes when they navigate and a query remounts past staleness. In this business the dashboard debt totals, credit-limit headroom (checked server-side at order create, orders.service.ts:131, but displayed client-side in NewOrder.tsx from cached client data), kassa balances, and order funnel are shared mutable state across admin/agents/cashier — all can be stale for 30s+ indefinitely while a page stays open.
REC: Cheapest first step: enable refetchOnWindowFocus and add refetchInterval (15-30s) on the dashboard/kassa/debts queries. Proper fix for the ERP rearchitecture: an SSE endpoint (no extra dep, works through the existing Express platform) or @nestjs/websockets gateway broadcasting entity-change events that trigger targeted query invalidation.

### [medium] salesTrend loads the entire order history and aggregates per-day in JS
apps/api/src/dashboard/dashboard.service.ts:102
salesTrend() does findMany over ALL recognized orders (select date/saleTotal/profit, no date window, no limit) then builds a per-day Map in JS. Every dashboard load transfers the full order history; payload and CPU grow linearly forever. Prisma groupBy can't group by day, but this belongs in SQL (strftime/date_trunc GROUP BY). Related JS-aggregation-over-findMany hotspots: clients.service.ts:37-40, factories.service.ts:38-42, and agents.service.ts:47-49 reduce over full per-party order/payment lists to compute totals that order.aggregate/groupBy already computes elsewhere in the same codebase.
REC: Use $queryRaw with GROUP BY date (and bound it to a rolling window, e.g. last 90 days, matching what the chart displays); replace detail-endpoint JS reduces with prisma.aggregate filtered by party id.

### [medium] Every mutation calls qc.invalidateQueries() with no key — global refetch storm of unbounded endpoints
apps/web/src/pages/Orders.tsx:23
17 occurrences across 13 page files (Orders, Payments, Products, Expenses, Clients, Agents, Factories, Vehicles, NewOrder, Import, and the detail pages) call queryClient.invalidateQueries() with no filter after a mutation. This marks every cached query stale and refetches all actively mounted ones. Combined with the unbounded API endpoints, a single status-advance click on the Orders page re-downloads the full orders list (all relations) plus any other mounted queries; on Import success it refetches everything at once.
REC: Invalidate by key: qc.invalidateQueries({ queryKey: ['orders'] }) plus the affected derived keys (['dashboard'], ['debts'], ['kassa']). Consider a small helper mapping mutation type to affected keys.

### [medium] No route-level code splitting — all 21 pages plus recharts and framer-motion ship in one bundle
apps/web/src/App.tsx:6
App.tsx statically imports all 21 page components (lines 6-27); grep confirms zero React.lazy/Suspense usage in apps/web/src. Dashboard pulls recharts and every page pulls framer-motion, so the login screen pays for the entire app including chart and animation libraries. On the low-bandwidth connections typical for field agents in-region, initial load is the whole ERP.
REC: Convert routes to React.lazy(() => import('./pages/X')) with a Suspense fallback inside Layout; recharts then splits naturally with the Dashboard/Reports chunks. Vite handles the rest with no config.

### [low] Payment and Expense lack a date index while their list endpoints sort by date desc over full-table scans
apps/api/prisma/schema.prisma:191
Payment has indexes on all FKs and type but no @@index([date]); payments.findAll (payments.service.ts:35-38) does an unbounded findMany orderBy date desc — a full scan + sort per request. Expense (schema line 238-251) likewise has no date index for expenses.service.ts:9. Order has single-column @@index([status]) and @@index([date]) but the common list query filters status/agentId AND sorts by date, where composites like ([agentId, date]) or ([status, date]) would serve better; the ubiquitous status != 'CANCELLED' predicate can't use the status index at all (negation). Low severity today because everything is a full scan anyway and SQLite tables are small, but these become the load-bearing indexes once pagination (take/skip with orderBy date) is added.
REC: Add @@index([date]) to Payment and Expense, and composite @@index([agentId, date]) / @@index([clientId, date]) on Order and Payment alongside the server-pagination work.

### [low] Dashboard summary runs its non-agent branch queries sequentially
apps/api/src/dashboard/dashboard.service.ts:68
In summary(), the admin branch awaits debts.summary(), then cashByCurrency('UZS'), then cashByCurrency('USD'), then the expense aggregate strictly in sequence (lines 68-75) — four independent awaits, each itself multi-query (debts.summary alone is ~12 queries including 3 recognizedPaymentWhere scans). cashByCurrency also does a two-step findMany(cashbox ids) + groupBy that could be a single groupBy filtered via the relation. Adds serial latency to the most-viewed endpoint.
REC: Wrap the four in Promise.all, and let cashByCurrency groupBy cashTransaction with where: { cashbox: { currency } } to drop the id-collection roundtrip.
