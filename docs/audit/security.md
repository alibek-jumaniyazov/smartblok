# security

## Domain summary
SmartBlok API = NestJS 10 + Prisma 5 (SQLite dev, Postgres-ready). Auth = Passport JWT (Bearer) + global-ish per-controller `@UseGuards(JwtAuthGuard, RolesGuard)`; no global APP_GUARD, so any controller/handler that omits the guard is public. Roles: ADMIN, ACCOUNTANT, AGENT, CASHIER. RolesGuard (roles.guard.ts:14) FAILS OPEN — when no `@Roles` metadata is present it returns true, so handlers with no `@Roles` are reachable by ANY authenticated user. JWT secret resolved via requireJwtSecret (common/jwt-secret.ts) which only enforces length>=16 (no check that it is not the committed dev value). Passwords: bcryptjs, 10 rounds; login has a DUMMY_HASH constant-time compare (good anti-enumeration). No `$queryRaw`/`$executeRaw` anywhere (no raw-SQL injection surface). No helmet, no @nestjs/throttler (no rate limiting). CORS from CORS_ORIGIN env, credentials:true. ValidationPipe whitelist+transform is global, but every controller takes `@Body() d: any` (bypasses DTO whitelisting for writes) and services hand-whitelist fields.

CONDENSED ENDPOINT -> GUARD / ROLES / OWNERSHIP TABLE (issues flagged):
- POST /auth/login — PUBLIC (intended) — no rate limit [FINDING].
- GET/PUT /auth/me — Jwt — self only; updateProfile CANNOT set role/agentId (no self-escalation — SAFE).
- agents.controller: ctrl has Jwt+RolesGuard but NO ctrl-level @Roles. GET /agents, GET /agents/:id => ANY authed incl AGENT [FINDING: leaks every rival agent's clients+sales+profit — same data the dashboard was locked down for]. POST/PUT=ADMIN,ACCOUNTANT; DELETE=ADMIN.
- clients.controller: @Roles(ADMIN,ACCOUNTANT,AGENT). GET / => scoped by agentId (OK). GET /:id => service.findOne(id) NO user, NO ownership [IDOR FINDING]. POST => forces agentId for AGENT (OK). PUT /:id => NO user, NO ownership + agentId & creditLimit mass-assignable [IDOR + credit-limit-bypass FINDING]. DELETE=ADMIN.
- orders.controller: @Roles(ADMIN,ACCOUNTANT,AGENT); every handler calls assertOwnership + scope (SAFE). Client-supplied salePricePerUnit/costPricePerUnit accepted with no floor [financial-trust FINDING].
- payments.controller: @Roles(+CASHIER); findAll scoped, create restricts AGENT to CLIENT type + own client, delete=ADMIN,ACCOUNTANT (mostly SAFE); dto.orderId not validated to belong to client [minor].
- debts=ADMIN,ACCOUNTANT; reports=ADMIN,ACCOUNTANT; kassa=ADMIN,ACCOUNTANT,CASHIER; expenses=ADMIN,ACCOUNTANT,CASHIER; users=ADMIN; import=ADMIN,ACCOUNTANT (file upload, no size/type limit); dashboard=ADMIN,ACCOUNTANT,AGENT (scoped, SAFE).
- factories/products/vehicles/regions controllers: Jwt+RolesGuard but NO ctrl-level @Roles, so all GET handlers => ANY authed incl AGENT [FINDING: product costPrice + factory/vehicle debt balances leak to agents]; mutations are ADMIN/ACCOUNTANT.
- procurement: matrix=ADMIN,ACCOUNTANT,AGENT; prices/routes=ADMIN,ACCOUNTANT.

Committed secret: apps/api/.env IS git-tracked (`git ls-files` confirms) and `.gitignore` force-adds it via `!apps/api/.env`; it contains JWT_SECRET="dev-secret-change-me-please-long-random" (39 chars, passes the length-only guard). No committed .env with real prod creds beyond this dev secret. Role escalation via self-service profile is NOT possible (auth.service.updateProfile whitelists name/username/email/phone/password only).

## Findings
### [high/CONFIRMED] Committed JWT secret passes the length-only guard, enabling ADMIN token forgery in production
apps/api/.env:2
apps/api/.env is git-tracked (confirmed via `git ls-files apps/api/.env`) and `.gitignore` deliberately force-includes it with `!apps/api/.env`. It sets JWT_SECRET="dev-secret-change-me-please-long-random". requireJwtSecret (apps/api/src/common/jwt-secret.ts:7) only rejects secrets shorter than 16 chars — this value is 39 chars, so it passes silently. Anyone with read access to this GitHub repo knows the signing key; if the app is deployed without overriding JWT_SECRET, an attacker forges a JWT with {sub, role:'ADMIN'} and gains full admin (users, financials, deletes). The guard gives a false sense of safety because it validates length, not that the value is non-public.
EVIDENCE: git ls-files apps/api/.env => "apps/api/.env" (tracked; committed in 1459789). .gitignore:12 "`.env`" ignored, then .gitignore:35 "`!apps/api/.env`" force-includes it. apps/api/.env:2 => `JWT_SECRET="dev-secret-change-me-please-long-random"` (printf|wc -c = 39 chars). apps/api/src/common/jwt-secret.ts:7 => `if (s.length < 16) { throw ... }` — length-only guard; 39 >= 16 so it passes silently, never checking the value is non-public. Secret is reused for signing (auth.module.ts:19 `secret: requireJwtSecret(config)`) and verifying (jwt.strategy.ts:13 `secretOrKey: requireJwtSecret(config)`). Role is trusted straight from the token: jwt.strategy.ts:17-25 `validate(payload) { return { ... role: payload.role, ... } }` with no DB re-check, and roles.guard.ts:16 `if (user && required.includes(user.role)) return true;`. Thus a JWT forged with the committed secret and `{sub, role:'ADMIN'}` passes JwtAuthGuard + RolesGuard for @Roles('ADMIN') endpoints.
REC: Remove apps/api/.env from git (git rm --cached), drop the `!apps/api/.env` allowlist line, rotate JWT_SECRET, and extend requireJwtSecret to reject known dev constants (e.g. any value containing 'change-me'/'dev-secret') at least when NODE_ENV==='production'.

### [high/CONFIRMED] IDOR: any AGENT can read any client's full financial detail via GET /clients/:id
apps/api/src/clients/clients.controller.ts:14
`@Get(':id') findOne(@Param('id') id) { return this.service.findOne(id); }` does not pass the CurrentUser, and ClientsService.findOne (apps/api/src/clients/clients.service.ts:30) has zero ownership scoping — it returns the client plus ALL orders, payments, balances and creditLimit. The controller allows AGENT (@Roles ADMIN,ACCOUNTANT,AGENT). While findAll IS scoped to the agent's own agentId, findOne is not, so an agent can enumerate UUIDs (or read ids leaked by GET /agents) and view a rival agent's clients' complete order/payment history and debts.
EVIDENCE: apps/api/src/clients/clients.controller.ts:9 `@Roles('ADMIN', 'ACCOUNTANT', 'AGENT')` (class-level, no override on findOne); :14 `@Get(':id') findOne(@Param('id') id: string) { return this.service.findOne(id); }` — CurrentUser not passed. apps/api/src/clients/clients.service.ts:30-41 `findOne(id)` uses `this.prisma.client.findUnique({ where: { id }, include: { agent: true, region: true } })` with no agentId scoping and returns `{ ...client, orders, payments, totals: { delivered, paid, balance, ordersCount } }` — full order history (incl. product/factory/vehicle), all CLIENT payments, balance, and the client record incl. creditLimit (field set at :48). Contrast: :9-11 `private scope(user) { return user?.role === 'AGENT' && user?.agentId ? { agentId: user.agentId } : {}; }` and :13-16 findAll(user) IS scoped, exactly as claimed. No mitigation exists: apps/api/src/auth/roles.guard.ts:14-17 checks role only ("if (!required || required.length === 0) return true; ... required.includes(user.role)"); main.ts registers only a ValidationPipe; grep across apps/api/src for middleware/interceptors/APP_GUARD finds only FileInterceptor (import.controller.ts). Enumeration vector verified: apps/api/src/agents/agents.controller.ts has NO class-level @Roles and no @Roles on GET routes (:11-12), so any authenticated AGENT passes RolesGuard (empty-required => true) and GET /agents/:id (agents.service.ts:30-58) returns a rival agent's client ids (:37 `agent.clients.map((c) => c.id)`) and full orders with client included (:41) — the leak path is real and even broader than claimed.
REC: Pass @CurrentUser() into findOne and add an assertOwnership check (AGENT may only read clients where client.agentId === user.agentId), mirroring OrdersService.assertOwnership.

### [high/CONFIRMED] IDOR + credit-limit bypass: any AGENT can update any client, steal it, and raise its credit limit via PUT /clients/:id
apps/api/src/clients/clients.controller.ts:16
`@Put(':id') update(@Param('id') id, @Body() d) { return this.service.update(id, d); }` passes no user, and ClientsService.update (apps/api/src/clients/clients.service.ts:51-61) has no ownership check and copies attacker-controlled `d.agentId` (line 59) and `d.creditLimit` (line 60) straight into the row. An AGENT can therefore (a) reassign ANY client — including a rival's — to their own agentId (client theft, redirecting future commissions/ownership), and (b) set creditLimit arbitrarily high, defeating the owner's mandated credit-limit enforcement in OrdersService.create (which reads client.creditLimit). This directly undermines a documented business rule.
EVIDENCE: apps/api/src/clients/clients.controller.ts:9 `@Roles('ADMIN', 'ACCOUNTANT', 'AGENT')` (controller-level; only @Delete at line 17 is re-scoped to ADMIN) — AGENT can call PUT. Line 16: `@Put(':id') update(@Param('id') id: string, @Body() d: any) { return this.service.update(id, d); }` — no @CurrentUser passed, body typed `any`. apps/api/src/clients/clients.service.ts:51-63 `update(id, d)` has no ownership/scope check and writes line 59 `...(d.agentId !== undefined ? { agentId: d.agentId || null } : {})` and line 60 `...(d.creditLimit !== undefined ? { creditLimit: Number(d.creditLimit) || 0 } : {})` straight into prisma.client.update. The AGENT scoping helper (lines 9-11 `scope(user)`) is applied ONLY in findAll (line 14), not in findOne/update/remove. No mitigation exists: apps/api/src/main.ts:13-18 registers ValidationPipe({ whitelist: true }) but the handler body is `any` (no DTO metatype), so nothing is stripped; roles.guard.ts only checks role membership; grep found no other APP_GUARD/APP_PIPE/interceptor. Business impact confirmed: apps/api/src/orders/orders.service.ts:131-133 enforces the credit limit by reading `client.creditLimit` with `creditLimit === 0` treated as UNLIMITED, so the attacker can even send creditLimit: 0 (or junk — `Number(d.creditLimit) || 0`) to disable enforcement entirely; and orders.service.ts:110-115 blocks AGENTs from booking for other agents' clients, which client reassignment via this endpoint bypasses.
CORRECTED: Claim fully accurate; two minor refinements: (1) the service update method spans lines 51-63, not 51-61 (cited field lines 59/60 are exact); (2) the impact is worse than stated — because OrdersService treats creditLimit 0 as unlimited and the service coerces via `Number(d.creditLimit) || 0`, an AGENT can set creditLimit to 0/garbage to remove the limit entirely, not just raise it.
REC: Pass @CurrentUser() into update, block AGENT from editing clients they don't own, and strip agentId/creditLimit from the update payload for non-ADMIN/ACCOUNTANT roles.

### [high/CONFIRMED] Broken access control: GET /agents and GET /agents/:id readable by any authenticated user (leaks all agents' clients and financials)
apps/api/src/agents/agents.controller.ts:11
AgentsController has @UseGuards(JwtAuthGuard, RolesGuard) but NO controller-level @Roles, and the GET handlers (`@Get() findAll`, `@Get(':id') findOne`) carry no @Roles either. Because RolesGuard (apps/api/src/auth/roles.guard.ts:14) returns true when no roles metadata exists, ANY authenticated user — including a low-trust AGENT — can call these. AgentsService.findOne (agents.service.ts:30) returns that agent's full client list, orders, sales, profit, collected and outstanding balances. This is exactly the company-wide/rival data the DashboardController was explicitly locked down to hide (see comment in dashboard.controller.ts), but it leaks unfiltered here. The same pattern exposes GET /factories, /products (costPrice/margins), and /vehicles to AGENT.
EVIDENCE: 1) apps/api/src/agents/agents.controller.ts:7-12 — class has guards but no @Roles, and both GET handlers are bare: `@UseGuards(JwtAuthGuard, RolesGuard)` / `@Controller('agents')` ... `@Get() findAll() { return this.service.findAll(); }` / `@Get(':id') findOne(@Param('id') id: string) { return this.service.findOne(id); }`. Only POST/PUT/DELETE carry @Roles('ADMIN','ACCOUNTANT') / @Roles('ADMIN') (lines 13-15).
2) apps/api/src/auth/roles.guard.ts:14 — `if (!required || required.length === 0) return true;` — with no roles metadata on handler or class, RolesGuard passes any authenticated user, so JwtAuthGuard alone gates the GETs.
3) AGENT logins exist and authenticate normally: apps/api/src/agents/agents.service.ts:78 creates `role: 'AGENT'` users; apps/api/src/auth/jwt.strategy.ts:17-24 validates any JWT and returns `role: payload.role` with no role filtering.
4) apps/api/src/agents/agents.service.ts:30-58 — findOne includes `clients: { orderBy: { name: 'asc' } }` (line 33), all orders with `include: { client: true, product: true, factory: true, vehicle: true }` (line 41), and returns `{ ...agent, orders, totals: { sales, profit, collected, outstanding, advance, ... } }` (line 58). findAll (lines 10-27) returns every agent annotated with `sales`, `profit`, `collected`. No agentId scoping or user parameter anywhere in the service.
5) The contradiction with the intended policy is explicit: apps/api/src/dashboard/dashboard.controller.ts:8-10 comment — "Previously guarded only by JwtAuthGuard — any authenticated user could read company-wide financials and every agent's numbers. Now role-guarded, and the service scopes an AGENT to their own agentId ... so nothing global leaks to a low-privilege login." GET /agents/:id returns exactly that data unscoped.
6) Same pattern confirmed elsewhere: apps/api/src/factories/factories.controller.ts:11-12 and apps/api/src/vehicles/vehicles.controller.ts:11-12 (bare @Get findAll/findOne); apps/api/src/products/products.controller.ts:11 (bare @Get findAll) with apps/api/src/products/products.service.ts:10 doing `prisma.product.findMany` with no `select`, returning all fields including costPrice (field confirmed at products.service.ts:20), so an AGENT can derive dealer margins.
7) No mitigations found: grep for APP_GUARD|APP_INTERCEPTOR|useGlobalGuards|ClassSerializerInterceptor over apps/api/src returned no matches; apps/api/src/main.ts:13-19 registers only a ValidationPipe (input-side, no output filtering).
CORRECTED: Claim is accurate as stated. Minor precision note: products.controller.ts has no @Get(':id') findOne (only @Get() findAll), so "GET /products" is exposed but there is no per-product GET; factories and vehicles expose both findAll and findOne unprotected.
REC: Add controller-level @Roles('ADMIN','ACCOUNTANT') to AgentsController (and factories/products/vehicles read endpoints, or scope them), and change RolesGuard to fail closed when a route under RolesGuard has no roles, so missing @Roles never silently grants access.

### [medium] RolesGuard fails open when no @Roles metadata is present
apps/api/src/auth/roles.guard.ts:14
`if (!required || required.length === 0) return true;` means any handler placed under RolesGuard but without a @Roles decorator is accessible to every authenticated user regardless of role. This is the root cause that turns the missing @Roles on agents/factories/products/vehicles GET routes into a data leak. It is a fragile default: any future endpoint added without @Roles is silently world-readable to all logged-in roles.
REC: Consider failing closed (deny when no roles specified on a RolesGuard-protected route) or require an explicit @Public()/@AllowAllRoles() marker so the absence of @Roles is a deliberate choice, not an accident.

### [medium] Order sale/cost prices trusted from client — agent can book arbitrary prices, under-report debt and bypass credit limit
apps/api/src/orders/orders.service.ts:124
create() computes totals from client-supplied dto.salePricePerUnit/costPricePerUnit (numOr falls back to product defaults only when the field is empty). An AGENT can send salePricePerUnit far below the real price: saleTotal drops, so (a) the recognized client debt is under-recorded, (b) the credit-limit check at line 131-137 (which uses this same saleTotal) is trivially bypassed by pricing the order low, and (c) reported profit is distorted. There is no validation that sale>=cost or that price matches the procurement matrix. update() (line 193) has the same exposure.
REC: For AGENT role, force salePricePerUnit/costPricePerUnit to the product's configured/matrix price (ignore client overrides), or validate the override against an allowed range and require ACCOUNTANT/ADMIN to deviate.

### [medium] No rate limiting or lockout on POST /auth/login (brute-force)
apps/api/src/auth/auth.controller.ts:11
The login route is public and there is no @nestjs/throttler / ThrottlerGuard anywhere (not in main.ts, app.module.ts, or the controller), and no failed-attempt lockout. The DUMMY_HASH constant-time compare mitigates username enumeration but does nothing against automated password brute-forcing. With bcrypt(10) an attacker can still run large credential-stuffing campaigns unthrottled.
REC: Add @nestjs/throttler with a strict limit on the login route (e.g. 5-10 attempts/min per IP+username) and consider progressive lockout on repeated failures.

### [low] No security headers (helmet) and permissive JSON handling in bootstrap
apps/api/src/main.ts:7
bootstrap() enables CORS and ValidationPipe but does not register helmet or any security-header middleware (HSTS, X-Content-Type-Options, frame options, etc.), and sets no request body size limit. For an ERP handling financial data this leaves the API without baseline hardening; combined with credentials:true CORS, a misconfigured CORS_ORIGIN would be more damaging.
REC: Add helmet(), set a JSON body size limit, and validate that CORS_ORIGIN is an explicit allowlist (never '*') in production.

### [low] Excel import accepts unbounded uploads with no size or MIME validation
apps/api/src/import/import.controller.ts:15
`@UseInterceptors(FileInterceptor('file'))` uses default Multer options — no fileSize limit and no MIME/extension filter. An authenticated ADMIN/ACCOUNTANT can upload an arbitrarily large file that is buffered fully in memory (file.buffer) and handed to the xlsx parser, enabling memory-exhaustion DoS or a decompression-bomb via a crafted xlsx.
REC: Configure FileInterceptor with limits.fileSize and a fileFilter restricting to .xlsx MIME types; stream or cap the parse.

### [low] Payment orderId not validated against the paying client
apps/api/src/payments/payments.service.ts:96
create() takes `orderId: dto.orderId || null` without verifying the order belongs to the client (clientId) the payment is booked against, or (for an AGENT) to the agent. An AGENT who is correctly restricted to their own client can still attach the payment to an unrelated order id, corrupting per-order reconciliation and the cancelled-order payment-exclusion logic in clients/factories/vehicles services.
REC: When dto.orderId is present, load the order and assert order.clientId === partyId (and order.agentId === user.agentId for AGENT) before writing.
