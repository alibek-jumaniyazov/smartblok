# db-schema

## Domain summary
Data layer of SmartBlok (gas-block dealer CRM/ERP). Single Prisma schema at apps/api/prisma/schema.prisma: 15 models — User, Agent, Region, Client, Factory, Product, Vehicle, FactoryPrice, LogisticsRoute, Order, Payment, Cashbox, CashTransaction, ExpenseCategory, Expense. All PKs are String UUIDs; the only business identifier is Order.orderNo (B-0001, @unique, generated in orders.service.ts by max-scan with collision retry). Provider is sqlite (dev.db present in apps/api/prisma/); docker-compose.yml provisions postgres:16-alpine but requires a manual provider switch. There is NO prisma/migrations directory — the workflow is `prisma db push` only (package.json scripts db:push / db:setup). Every monetary field (Product.costPrice/salePrice, FactoryPrice.pricePerM3/dealerBonusPct, LogisticsRoute.costPerTruck, Order.costPricePerUnit/salePricePerUnit/transportFee/costTotal/saleTotal/profit, Payment.amount/usdAmount/rate, CashTransaction.amount/rate, Expense.amount, Client.creditLimit) is Float. No Prisma enums — status/type/method/role/currency are free Strings validated only in NestJS services. No onDelete rules are declared anywhere, so Prisma defaults apply: Restrict on required FKs, SetNull on optional FKs (Payment.clientId/factoryId/vehicleId/agentId/orderId/cashboxId, Order.agentId/vehicleId, Client.agentId/regionId, Expense.categoryId/cashboxId are all optional). Order is single-line (one productId + quantity) and stores derived totals (costTotal, saleTotal, profit) plus a factoryId that duplicates product.factoryId. Payment is a polymorphic multi-party row (type CLIENT/FACTORY/VEHICLE, four nullable party FKs, optional orderId, optional cashboxId); posting to the ledger creates a CashTransaction whose paymentId/expenseId are plain Strings, not FK relations. Cashbox routing is by unique name via a CASHBOX_BY_METHOD map (seed.ts and payments service share it). Soft delete is partial: active Boolean exists on User/Agent/Factory/Product/Vehicle/Cashbox but not on Client/Region/Order/Payment/Expense/ExpenseCategory; no model has updatedAt, deletedAt, or createdBy/updatedBy. Order deletion is soft-cancel (status=CANCELLED) per orders.service.ts; debt recognition (apps/api/src/common/recognition.ts) counts all non-CANCELLED orders, matching the owner decision (debt at order creation) but contradicting docs/04 §4.5.1 which still says DELIVERED/COMPLETED only. Credit limit IS enforced in orders.service.ts (creditLimit>0 = cap), contradicting docs/04 §4.3.4 ("stored, not checked") — the doc is stale in both directions. seed.ts wipes all tables in FK order, then seeds 5 regions, 5 factories, 7 FactoryPrice rows, 5 routes, 4 products, 7 agents, 4 users (bcrypt, weak demo passwords), 13 clients (all creditLimit 100,000,000), 3 vehicles, 4 cashboxes, 10 orders, 10 payments (mirrored into CashTransaction), 5 expense categories, 3 expenses; client payments are never linked to orders (orderId always null). Missing enterprise entities: order line items, payment-to-order allocation, order status history/audit log, factory bonus wallet, pallet tracking, transport profit split, debt/opening-balance adjustments, documents/attachments, cashbox day-close, agent-level credit limits.

## Findings
### [critical/PARTIAL] All money fields are Float — unacceptable for a financial/ledger system
apps/api/prisma/schema.prisma:56
Every monetary column is Float (SQLite REAL / IEEE-754 double): Client.creditLimit, Product.costPrice/salePrice, FactoryPrice.pricePerM3, LogisticsRoute.costPerTruck, Order.costPricePerUnit/salePricePerUnit/transportFee/costTotal/saleTotal/profit, Payment.amount/usdAmount/rate, CashTransaction.amount/rate, Expense.amount. UZS amounts routinely reach 10^7–10^9 (seed has 100,000,000 credit limits, 5,000,000 truck costs); doubles hold integers exactly only up to 2^53, but derived values like qty 32.8 * 730000 already produce non-representable decimals, and debt = sum(saleTotal) - sum(payments) accumulates rounding drift, so client balances will show values like 23,943,999.999999996 and never reconcile to zero. docs/04 §4.7.2 itself flags this. Note Prisma does not support Decimal on the sqlite provider, so this finding is coupled to the provider finding.
EVIDENCE: CONFIRMED schema facts — apps/api/prisma/schema.prisma:6 `provider = "sqlite"`; :56 `creditLimit Float @default(0)`; :85-86 `costPrice Float` / `salePrice Float`; :112 `pricePerM3 Float`; :125 `costPerTruck Float`; :147-152 `costPricePerUnit Float`, `salePricePerUnit Float`, `transportFee Float @default(0)`, `costTotal Float @default(0)`, `saleTotal Float @default(0)`, `profit Float @default(0)`; :183-185 `usdAmount Float`, `rate Float`, `amount Float`; :218-219 `amount Float`, `rate Float` (CashTransaction); :243 `amount Float` (Expense). Seed magnitudes confirmed — apps/api/prisma/seed.ts:106 `creditLimit: 100000000`; :56 `costPerTruck: 5000000`. Docs flag confirmed — docs/04-malumotlar-modeli.md:14 "Pul maydonlari — `Float`... SQLite'da `REAL`" and :629 (§4.7 item 2) "yaxlitlash xatolari to'planishi mumkin". Unrounded storage confirmed — apps/api/src/orders/orders.service.ts:15-17 `const costTotal = quantity * cost; const saleTotal = quantity * sale; ... profit: saleTotal - costTotal - transport` (no rounding before persist); apps/api/src/payments/payments.service.ts:46 `if (method === 'USD') amount = usdAmount * rate;` (unrounded). REFUTED consequence — the auditor missed apps/api/src/common/recognition.ts:9-13: "// Round money to whole UZS so float artefacts (e.g. 32.8 * 750000 = 24,599,999.999…) do not leak phantom sub-tiyin debt/advance rows. export function roundMoney(n: number): number { return Math.round(n ?? 0); }" — applied on every party-balance read path: apps/api/src/clients/clients.service.ts:26 `balance: roundMoney(delivered - paid)`, debts.service.ts:35/40/45, factories.service.ts:24/42, vehicles.service.ts:21/36, agents.service.ts:48-56, dashboard.service.ts:24/41/79-84, reports.service.ts:26/33-37. Per-order drift ~1e-6 UZS sums far below 0.5, so displayed client balances round exactly and DO reconcile to zero. Residual unmitigated: apps/api/src/kassa/kassa.service.ts:14 `balance: inSum - outSum` and :65-66 (no rounding — kassa balances can show fractional artifacts from usdAmount*rate); orders.service.ts:133 credit-limit compares unrounded floats.
CORRECTED: The structural defect is real and fully verified: every monetary column is Float on the sqlite provider (where Prisma Decimal is unavailable), and order totals / USD-converted payment amounts are persisted as unrounded float products (orders.service.ts:15-17, payments.service.ts:46), so stored ledger rows do carry values like 23,943,999.999999996. However the specific claimed symptom — "client balances will show 23,943,999.999999996 and never reconcile to zero" — is wrong: the codebase deliberately mitigates it via roundMoney (Math.round to whole UZS, common/recognition.ts:11-13, whose comment cites this exact artifact), applied on all client/factory/vehicle/agent/debt/dashboard/report balance computations, so user-visible balances reconcile exactly. Residual gaps: kassa balances (kassa.service.ts:14,65-66) and the credit-limit comparison (orders.service.ts:133) use unrounded floats, and stored values remain imprecise — so the finding stands as a data-model debt (band-aid rounding at read time instead of exact storage), but overstated in its reconciliation consequence. The 2^53 point is technically true but immaterial at UZS 10^9–10^13 magnitudes.
REC: Move to PostgreSQL and change all money fields to Decimal @db.Decimal(18,2) (or store integer so'm as BigInt since UZS has no working subunit). Keep Float only for physical measures (quantity m3, truckCapacityM3). Recompute/backfill stored totals during migration and round rates to a fixed scale (e.g. Decimal(18,6) for FX rate).

### [critical/CONFIRMED] No migration history — schema managed only via `prisma db push`
apps/api/package.json:10
There is no apps/api/prisma/migrations directory; package.json only has db:push/db:setup scripts. For a production ERP this means: no versioned, reviewable DDL; no rollback; `db push` can silently drop columns/data on incompatible changes; and it is impossible to add anything Prisma can't express (CHECK constraints, partial indexes, triggers, SQL views) because there is no migration file to put raw SQL in. Combined with the pending sqlite→postgres switch, there is no defined path to move data.
EVIDENCE: 1) No migrations directory: Glob of apps/api/prisma/** returns only `schema.prisma`, `seed.ts`, `dev.db` — no `migrations/` folder exists. 2) apps/api/package.json lines 9-12: the only Prisma scripts are `"prisma:generate": "prisma generate"`, `"db:push": "prisma db push --skip-generate"`, `"db:setup": "prisma generate && prisma db push --skip-generate && tsx prisma/seed.ts"`, `"seed": "tsx prisma/seed.ts"` — no `prisma migrate` script anywhere. 3) Repo-wide grep for "migrate" (excluding node_modules) hits only documentation: docs/10-nofunksional-talablar.md:321 explicitly admits "Hozirda rasmiy migration skripti yoʻq — `prisma db push` (migrationsiz sxema push) ishlatiladi" ("There is currently no official migration script — prisma db push is used"), and lists adopting `prisma migrate` as future work (NFR-SCALE-05, RM-10, risk item 18). 4) Pending sqlite→postgres switch confirmed: apps/api/prisma/schema.prisma:6 `provider = "sqlite"`, while docker-compose.yml:1-3 says "Production/optional: PostgreSQL... To use Postgres, switch the provider in apps/api/prisma/schema.prisma to \"postgresql\"" and provisions postgres:16-alpine — with no migration files, there is no versioned DDL path to carry data across that switch. No compensating tooling (custom SQL scripts, migration framework, CI schema checks) exists elsewhere in the repo.
REC: Adopt `prisma migrate dev`/`prisma migrate deploy` now, baseline the current schema as migration 0, and gate schema changes through migrations in CI. Do the Postgres+Decimal switch as the first real migration while data volume is tiny.

### [high/CONFIRMED] Dev provider is SQLite while production targets Postgres — untested divergence
apps/api/prisma/schema.prisma:6
schema.prisma hardcodes provider = "sqlite"; docker-compose.yml ships postgres:16-alpine with a comment saying to hand-edit the provider. Prisma cannot parameterize provider, so dev and prod run different engines with different type mappings (no Decimal, no enums, no BigInt semantics on SQLite), different concurrency (SQLite single-writer will serialize/lock under multi-user cashier+agent load), and different constraint behavior. Nothing in CI exercises the Postgres path, so the 'switch later' plan will surface type and locking bugs at deploy time.
EVIDENCE: apps/api/prisma/schema.prisma:5-8: `datasource db { provider = "sqlite"; url = env("DATABASE_URL") }` — provider hardcoded to sqlite (Prisma requires a literal here; env() for provider is unsupported). docker-compose.yml:1-6: `# Dev default is SQLite (no setup). To use Postgres, switch the provider in # apps/api/prisma/schema.prisma to "postgresql" and set DATABASE_URL accordingly.` + `image: postgres:16-alpine` — the hand-edit switch plan exactly as claimed. CI: no .github directory exists (Glob '.github/**/*' returns no files); only project YAML is docker-compose.yml — there is no CI at all, so the Postgres path is never exercised. Migrations: apps/api/prisma/ contains only schema.prisma, seed.ts, dev.db (no migrations/ dir); apps/api/package.json:10-11: `"db:push": "prisma db push --skip-generate"` — schema sync is push-only, no migration history to validate on Postgres. Type divergence is materialized in the schema: zero Decimal/enum/BigInt occurrences; Float used 20 times (all money fields, e.g. schema.prisma:56 `creditLimit Float @default(0)`); roles/statuses are String (schema.prisma:16 `role String @default("AGENT") // ADMIN | ACCOUNTANT | AGENT | CASHIER`). Project docs admit the cause and risk: docs/02-arxitektura-va-texnologiyalar.md:202 (SQLite lacks Prisma enum, all types stored as String, validated only in service code), docs/04-malumotlar-modeli.md:14 and docs/10-nofunksional-talablar.md:236 INT-R6 (money as Float/SQLite REAL with acknowledged precision risk). Concurrency: apps/api/.env:1 `DATABASE_URL="file:./dev.db"` with no busy_timeout/connection_limit tuning and no write-serialization code — SQLite single-writer behavior unmitigated. No refuting guards found anywhere.
CORRECTED: One nuance, not a refutation: because the schema was already reduced to SQLite's lowest common denominator (Float money, String pseudo-enums, no BigInt), a provider swap would not break on those specific type mappings at migration time — the divergence has instead already degraded the eventual Postgres deployment (float currency arithmetic, DB-unenforced enums). Also, "nothing in CI exercises the Postgres path" understates it: the repo has no CI pipeline of any kind, and no Prisma migrations exist (push-only), so the switch path is doubly untested.
REC: Standardize on postgresql in schema.prisma for all environments and use the docker-compose db (or a testcontainer) for dev/CI. This also unblocks Decimal, native enums, and raw-SQL CHECK constraints.

### [high/PARTIAL] Default SetNull cascades silently corrupt financial history on hard deletes
apps/api/prisma/schema.prisma:171
No relation declares onDelete, so Prisma applies SetNull to every optional FK. Deleting a Client nullifies Payment.clientId on all their payments (a client with payments but no orders can be hard-deleted and their money history becomes orphaned 'phantom' payments); deleting an Agent nullifies agentId on Users, Clients, Orders and Payments, destroying historical sales attribution and agent debt/commission math; deleting a Vehicle nullifies Order.vehicleId/Payment.vehicleId; deleting a Cashbox nullifies Payment.cashboxId while its CashTransactions block on Restrict — asymmetric. docs/04 §4.4.1 claims 'no cascade — any delete of an object with dependents throws an FK error', which is wrong for every optional relation, so the documented safety property does not actually hold.
EVIDENCE: SCHEMA CLAIM CONFIRMED: grep for "onDelete" across apps/api/prisma returns zero matches; schema.prisma declares every relation bare, e.g. Payment.clientId "clientId  String?" / "client    Client?  @relation(fields: [clientId], references: [id])" (schema.prisma:173-174), Payment.agentId (171-172), Payment.vehicleId (177-178), Payment.cashboxId (187-188), Order.agentId (136-137), Order.vehicleId (144-145), User.agentId (19-20), Client.agentId (54-55). Prisma's default for optional relations is SetNull, required is Restrict, so at DB level the SetNull behavior exists exactly as claimed, and the Cashbox asymmetry is real in the schema: Payment.cashboxId optional (SetNull) vs CashTransaction.cashboxId required "cashboxId String" (schema.prisma:214, Restrict). Datasource is sqlite (schema.prisma:6), no migrations folder exists (db push). DOCS CLAIM CONFIRMED: docs/04-malumotlar-modeli.md:338 states "Kaskad o'chirish yo'q... Bog'liq yozuvi bo'lgan obyektni... o'chirishga urinish FK cheklovi tufayli DB xatosi beradi" and docs/07-api-spetsifikatsiyasi.md:739 "FK bogliqlik bolsa Prisma xatosi (500) berishi mumkin" — both wrong for every optional FK, which silently SetNulls instead of erroring. HOWEVER, THE CONCRETE CORRUPTION SCENARIOS ARE BLOCKED BY SERVICE-LAYER GUARDS THE AUDITOR MISSED: (1) clients.service.ts:64-74 — remove() counts orders AND payments and throws BadRequestException('Mijozda buyurtma yoki to'lov tarixi bor — o'chirib bo'lmaydi') if either > 0, so "a client with payments but no orders can be hard-deleted" is false via the API. (2) agents.service.ts:98-107 — remove() counts clients/orders/payments and throws before delete; user detach at :110 happens only for history-free agents. (3) vehicles.service.ts:45-53 — remove() counts orders/payments and throws. (4) factories.service.ts:52-61 — same guard pattern. (5) Cashbox has NO delete endpoint at all — kassa.controller.ts:13-15 exposes only GET/POST cashboxes, so the asymmetric Cashbox delete is unreachable. Residual genuinely-unguarded paths: regions.service.ts:10 (region.delete SetNulls Client.regionId — non-financial) and expenses.service.ts:52 removeCategory (expenseCategory.delete SetNulls Expense.categoryId, silently stripping categorization from financial expense history with no guard).
CORRECTED: The schema mechanics and the docs inaccuracy are real: no onDelete is declared anywhere, so Prisma applies SetNull to all optional FKs at the DB level, and docs/04 §4.4.1's "any delete with dependents throws an FK error" is false for every optional relation. But the claim's corruption scenarios are overstated: every delete endpoint for Client, Agent, Vehicle, and Factory has an explicit service-layer guard that counts dependent orders/payments (and clients/products where relevant) and throws BadRequestException before deleting — a client with payments but no orders is explicitly blocked at clients.service.ts:70-72 — and Cashbox has no delete endpoint at all, making the Payment/CashTransaction asymmetry unreachable through the API. The real residual defect is (a) the safety invariant lives only in application code, so any raw Prisma call, seed/import script, future endpoint, or direct SQLite access silently SetNulls financial FKs with no DB-level backstop (import.service.ts:42-47 already does bulk deleteMany), (b) the documentation asserts a DB-level guarantee that does not exist, and (c) two live endpoints remain genuinely unguarded: Region delete (nullifies Client.regionId) and ExpenseCategory delete (silently nullifies Expense.categoryId on financial expense records).
REC: Declare onDelete: Restrict explicitly on all party/ledger FKs (Payment.*, Order.*, CashTransaction.cashboxId) and reserve deletion for soft-delete flags. Fix docs/04 §4.4.1.

### [high/PARTIAL] Ledger linkage CashTransaction.paymentId/expenseId are not real foreign keys
apps/api/prisma/schema.prisma:222
paymentId and expenseId are bare String? columns with @@index but no @relation. The DB cannot guarantee a kassa posting points at an existing Payment/Expense, nothing prevents two CashTransactions claiming the same paymentId, and the reversal-on-delete flow (deleteMany({where:{paymentId}})) is purely conventional — a manual delete in the kassa module (docs/04 §4.7.3 confirms there is no barrier) desynchronizes cash balance from payments with no way to detect it. For the system of record of cash, referential integrity must be structural.
EVIDENCE: SCHEMA CLAIM CONFIRMED — apps/api/prisma/schema.prisma:222-228 (model CashTransaction): `paymentId String?  // set when source=PAYMENT — lets a deleted payment reverse its kassa entry` and `expenseId String?  // set when source=EXPENSE ...`, with only `@@index([paymentId])` / `@@index([expenseId])` — no @relation, no @@unique. So there is no DB-level FK, nothing structurally prevents a dangling paymentId or two rows sharing one paymentId. REVERSAL-IS-CONVENTIONAL CONFIRMED — apps/api/src/payments/payments.service.ts:119 `await tx.cashTransaction.deleteMany({ where: { paymentId: id } });` and apps/api/src/expenses/expenses.service.ts:45 `await tx.cashTransaction.deleteMany({ where: { expenseId: id } });` (both inside prisma.$transaction, so the paired delete itself is atomic). BUT THE CITED EXPLOIT PATH IS WRONG — the auditor's "manual delete in the kassa module ... no barrier" is refuted by current code: apps/api/src/kassa/kassa.service.ts:52-61 `async removeTransaction(id)` does `if (row.source !== 'MANUAL') { throw new BadRequestException("Bu yozuv to'lov yoki xarajatdan kelib chiqqan — uni shu yerdan o'chirib bo'lmaydi...") }` before delete; kassa.controller.ts:17 routes DELETE /transactions/:id only through this method, and grep shows no other cashTransaction.delete path in apps/api/src. The auditor relied on docs/04-malumotlar-modeli.md:630 ("kassa modulida shu yozuvni qo'lda o'chirishga to'siq yo'q" — no barrier to manual deletion), which is STALE: `git diff HEAD -- apps/api/src/kassa/kassa.service.ts` shows the guard was added in the uncommitted working tree, replacing HEAD's unguarded `removeTransaction(id) { return this.prisma.cashTransaction.delete(...) }`.
CORRECTED: The structural half of the claim is true: paymentId/expenseId in CashTransaction (schema.prisma:222-228) are plain indexed String? columns with no @relation or unique constraint, so referential integrity for the cash ledger is enforced only by application convention (deleteMany reversal inside $transaction in payments.service.ts:117-122 and expenses.service.ts:43-48). However, the claimed consequence — that a manual kassa delete can silently desync cash from payments "with no barrier" — is false against the current code: kassa.service.ts:52-61 explicitly rejects deletion of any CashTransaction whose source is not MANUAL, so PAYMENT/EXPENSE-sourced rows cannot be deleted via the kassa API; the only delete paths for those rows are the parent-entity remove() flows, which are transactional and drift-free. The docs/04 §4.7 statement of "no barrier" describes the committed HEAD version, not the working tree (the guard is an uncommitted change). Residual real exposure is narrower than claimed: integrity is app-level not DB-level (direct DB writes, future code paths, or a bug could still create dangling/duplicate paymentId references, and no reconciliation job exists to detect drift), and HEAD-as-committed does still lack the guard.
REC: Make them proper optional relations with onDelete: Restrict (delete the payment through the service which removes the posting first), add @@unique([paymentId]) / @@unique([expenseId]) if one posting per source is the invariant, or better: make Payment/Expense reference their CashTransaction and treat the ledger as append-only with explicit reversal rows instead of deletions.

### [high/CONFIRMED] No payment-to-order allocation model — client debt cannot be settled per order
apps/api/prisma/schema.prisma:179
Payment has a single optional orderId, and the seed never sets it — all client payments float at client level. There is no PaymentAllocation join table, so one payment cannot be split across several orders (the normal wholesale pattern: client pays 50M against 3 deliveries), per-order paid/outstanding status cannot be persisted (only recomputed by ad-hoc FIFO in recognition.ts, which excludes payments tied to cancelled orders via a notIn list — fragile), and aging reports per invoice are impossible. This blocks core ERP receivables functionality the rearchitecture needs.
EVIDENCE: 1) Single optional orderId, no allocation table — apps/api/prisma/schema.prisma:179-180: `orderId   String?` / `order     Order?   @relation(fields: [orderId], references: [id])`. Case-insensitive grep for `PaymentAllocation|allocation` across apps/api returns zero matches; there is no join/allocation model, so one payment cannot be split across multiple orders.

2) Seed never sets orderId — apps/api/prisma/seed.ts:157-163, the `pay()` helper creates payments with only `clientId/factoryId/vehicleId/agentId/cashboxId`; grep for `orderId` in seed.ts: no matches. The web frontend also never sends it — grep for `orderId` in apps/web/src: no matches — so in practice ALL client payments float at client level, even though payments.service.ts:96 (`orderId: dto.orderId || null`) would accept one via API.

3) Fragile notIn exclusion — apps/api/src/common/recognition.ts:21-27: `recognizedPaymentWhere` loads ALL cancelled order ids (`prisma.order.findMany({ where: { status: 'CANCELLED' }, select: { id: true } })`) then builds `{ OR: [{ orderId: null }, { orderId: { notIn: ids } }] }` — an unbounded in-memory id list injected into every balance query, exactly as claimed. Used in 10+ services (debts, clients, agents, factories, vehicles, dashboard, reports, orders credit check).

4) No per-order paid/outstanding anywhere — all receivables math is aggregate per party: apps/api/src/debts/debts.service.ts:20-45 (`groupBy(['clientId']) _sum.saleTotal` minus `groupBy` of payments → `balance: roundMoney(delivered - paid)`), same pattern in clients.service.ts:15-18 and orders.service.ts:74-79 (credit-limit check). orders.service.ts:56 includes `payments: true` on findOne but nothing computes settlement from it. Per-invoice aging is impossible with this data model.

CORRECTION to one claim detail (does not weaken the finding): recognition.ts contains NO "ad-hoc FIFO" — grep for `FIFO|fifo` in apps/api/src: zero matches. Per-order paid/outstanding is not recomputed at all, by FIFO or otherwise; recognition.ts only filters out payments linked to cancelled orders. The defect is therefore slightly WORSE than stated: there is no per-order settlement logic of any kind.
REC: Add PaymentAllocation { id, paymentId FK, orderId FK, amount Decimal, createdAt, @@unique([paymentId, orderId]) } and keep Payment.orderId only as a legacy hint or drop it. Enforce sum(allocations.amount) <= payment.amount in the service layer.

### [high/CONFIRMED] No audit trail: no updatedAt, no createdBy/actor attribution, no order status history
apps/api/prisma/schema.prisma:132
Not a single model has @updatedAt; Orders, Payments, Expenses and CashTransactions do not record which User created or modified them — in a cash business with a CASHIER role, money movements are unattributable. Order.status transitions overwrite in place with no OrderStatusHistory/timeline table and no deliveredAt/completedAt/cancelledAt timestamps, so 'when was it delivered', cycle-time KPIs, and dispute resolution ('who cancelled this order and when') are unanswerable. The owner decision 'delete = soft-cancel with reversal' is implemented but leaves zero trace of who/when/why.
EVIDENCE: 1) No @updatedAt anywhere: grep for "updatedAt|createdBy|audit|History|deliveredAt|completedAt|cancelledAt" across apps/api/prisma/schema.prisma returns zero matches (all apparent hits in src/ were the substring "actor" inside "Factory"). Every model carries only `createdAt DateTime @default(now())` — e.g. schema.prisma:155 (Order), :189 (Payment), :224 (CashTransaction), :247 (Expense); Region (:39-45), ExpenseCategory (:232-236) and FactoryPrice (:107-117) have no timestamps at all beyond effectiveFrom.

2) No actor attribution: Order (schema.prisma:132-165), Payment (:167-198), CashTransaction (:212-230), Expense (:238-251) have no userId/createdBy field, and model User (:10-24) has no back-relations to any of them. The JWT user reaches services but is used ONLY for authorization, never persisted: orders.service.ts:29-38 (`scope()`/`assertOwnership()`), payments.service.ts:73-83 (AGENT restrictions). kassa.service.ts and expenses.service.ts contain zero occurrences of "user" — a CASHIER's manual CashTransaction or Expense records no identity whatsoever. Payment.agentId/Order.agentId identify the sales Agent business entity (forced to self only for AGENT-role users, orders.service.ts:110-111), not the User account, and are absent/arbitrary for ADMIN/ACCOUNTANT/CASHIER-created records.

3) Status overwritten in place, no history/timestamps: orders.service.ts:215 `return this.prisma.order.update({ where: { id }, data: { status } });` (setStatus), :227 same in advance(), and remove() at :233-239 does `data: { status: 'CANCELLED' }` — no OrderStatusHistory model exists in the schema, no deliveredAt/completedAt/cancelledAt columns, no who/when/why recorded on cancel. schema.prisma:153 confirms status is a single mutable string: `status String @default("NEW") // NEW|CONFIRMED|...|CANCELLED`.

4) No compensating infrastructure: grep for Interceptor/Logger/Middleware in apps/api/src matches only FileInterceptor for Excel upload (import.controller.ts:2,15); apps/api/src/common/ contains only recognition.ts and jwt-secret.ts. Datasource is sqlite (schema.prisma:6), so no DB-level audit either.

REC: Add updatedAt DateTime @updatedAt everywhere; add createdById (FK User) to Order, Payment, Expense, CashTransaction; add OrderEvent { orderId, fromStatus, toStatus, userId, at, note } written on every transition; optionally a generic AuditLog for entity mutations.

### [high/CONFIRMED] Soft-delete is inconsistent: Client, Order-adjacent finance rows and Region have no active/deletedAt at all
apps/api/prisma/schema.prisma:47
active Boolean exists on User/Agent/Factory/Product/Vehicle/Cashbox but Client — the entity most likely to churn — has no active or deletedAt, and neither do Region, ExpenseCategory, Payment or Expense. Combined with the SetNull cascade finding, the only ways to retire a client are hard delete (corrupts payments) or leaving them forever in pickers. docs/04 §4.7.7 also notes active is mostly not honored as a filter, so even where the flag exists it doesn't produce soft-delete semantics.
EVIDENCE: SCHEMA (apps/api/prisma/schema.prisma) — field distribution is exactly as claimed. ### [medium] Order.factoryId duplicates Product.factoryId with no consistency guarantee
apps/api/prisma/schema.prisma:140
Order stores both factoryId and productId, but Product already belongs to exactly one Factory (@@unique([factoryId, name])). Nothing prevents creating an order whose factoryId differs from product.factoryId, which would mis-post factory debt (costTotal is 'we owe factory') to the wrong factory while the product came from another. Same class of duplication: Order.agentId and Payment.agentId duplicate Client.agentId (defensible as historical snapshot, but undocumented as such).
REC: Either drop Order.factoryId and derive via product, or keep it as a denormalized read-optimization and enforce order.factoryId === product.factoryId in the service plus a DB CHECK/trigger once on Postgres migrations. Document agentId-on-order as an intentional snapshot.

### [medium] Order model is single-line — no OrderItem table for multi-product orders
apps/api/prisma/schema.prisma:142
An order is exactly one product with one quantity/price. Real gas-block trucks routinely mix sizes (seed itself has 600x300x200 and 600x300x100 products); today that forces two Order rows with two orderNos, two transportFees for one physical truck, and doubled logistics cost math (LogisticsRoute.costPerTruck can't be applied to a mixed load). For the enterprise rearchitecture this is a structural blocker.
REC: Introduce OrderItem { orderId, productId, quantity, costPricePerUnit, salePricePerUnit } and move per-line snapshots there; keep order-level transportFee, totals, and status. Migrate existing orders as single-item orders.

### [medium] Statuses/types/roles are free Strings with no DB-level constraint
apps/api/prisma/schema.prisma:153
Order.status, Payment.type/method, Cashbox.type/currency, CashTransaction.direction/source, User.role, FactoryPrice.paymentMethod are all String with allowed values only in comments. Any typo ('DELIVRED', 'Cash') silently corrupts financial aggregation because recognition/debt queries filter by exact string (status not 'CANCELLED', type === 'CLIENT'). The schema comment for Payment.method is already out of sync with reality (service accepts TRANSFER; comment lists 5 values) — docs/04 §4.5.3 documents this drift.
REC: After the Postgres switch, convert these to Prisma enums (or CHECK constraints via migration SQL). Until then, centralize allowed-value constants in one shared module (apps/api/src/common) and validate in DTOs, and fix the Payment.method comment to include TRANSFER.

### [medium] Payment lacks party/type consistency constraints — a payment can point at everyone or no one
apps/api/prisma/schema.prisma:167
Payment.type says CLIENT/FACTORY/VEHICLE but clientId, factoryId and vehicleId are independent nullable FKs: a row can have type=CLIENT with only factoryId set, all three set, or none set, and debt aggregation would then miscount or drop it. There is also no CHECK amount > 0, no rule that method=USD requires usdAmount and rate > 0 (both default 0, so amount=usdAmount*rate can silently be 0), and no receipt/document number for cashier traceability.
REC: On Postgres, add CHECK constraints via migration: exactly-one-party matching type, amount > 0, (method='USD') => (usdAmount>0 AND rate>0). Add a paymentNo business identifier. Longer-term, model counterparty as a single partyType+partyId or a Counterparty table instead of four nullable FKs.

### [medium] FactoryPrice and LogisticsRoute allow duplicate/ambiguous effective rows and miss lookup indexes
apps/api/prisma/schema.prisma:107
FactoryPrice has no @@unique([factoryId, paymentMethod, effectiveFrom]) and only @@index([factoryId]) — two identical rows for the same factory/method/date are legal, making 'current price' nondeterministic; the natural query (latest effectiveFrom <= orderDate per factory+method) has no supporting index (factoryId, paymentMethod, effectiveFrom). LogisticsRoute similarly lacks @@unique([factoryId, regionId, effectiveFrom]); its composite @@index([factoryId, regionId]) covers factory-first lookups but leaves regionId unindexed for region-side queries and Postgres FK checks.

### [medium] Missing indexes on date fields used by every report: Payment.date, Expense.date, CashTransaction.date
apps/api/prisma/schema.prisma:169
Order has @@index([date]) but Payment, Expense and CashTransaction — the rows dashboards, kassa screens and period reports filter by date range constantly — have no date index. ### [medium] Missing business entities: bonus wallet, pallet tracking, transport profit, debt adjustments, documents, cash day-close
apps/api/prisma/schema.prisma:1
Gaps versus the business the ERP must run: (1) FactoryPrice.dealerBonusPct exists but there is no BonusTransaction/wallet to accrue, track and redeem factory bonuses — the 3–5% seeded bonus money is invisible; (2) no Pallet/PalletBalance model though gas blocks ship on returnable pallets (per-client and per-factory pallet debt is standard in this trade); (3) transportFee is a single dealer-absorbed cost — no split between transport charged to client vs paid to carrier, so transport can never become a profit line and Vehicle payables can't be reconciled per-trip; (4) no DebtAdjustment/OpeningBalance entity — importing legacy balances (import module exists) has nowhere clean to land except fake orders/payments; (5) no Document/Attachment model for waybills and invoices; (6) no CashboxClosing (day-close with counted-vs-expected) for cashier reconciliation; (7) creditLimit exists only on Client — no Agent-level exposure limit even though agents front the credit chain.

### [medium] Currency modeling is ambiguous: USD cashbox balances and CashTransaction.amount currency undefined

Cashbox has currency UZS|USD, but CashTransaction.amount has no currency field — for the 'Naqt kassa (USD)' box it is unclear whether amount is UZS-equivalent or USD; Payment stores usdAmount+rate+amount with all defaults 0, and the seed posts payment.amount (UZS) into whatever box the method maps to. ### [low] Stored derived totals (costTotal, saleTotal, profit) can drift from their inputs
